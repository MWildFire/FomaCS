Операционные и управляющие системы

Тема A1.3 — про два типа «дирижёров» в мире вычислений. Операционная система (OS) управляет ресурсами компьютера: процессорным временем, памятью, дисками, сетью, безопасностью. Управляющая система (control system) регулирует поведение физического устройства — лифта, термостата, машины — через датчики, контроллер, актуаторы и обратную связь. Внутри одна и та же идея: скрыть сложность и принять решение на основе данных. Разделы A1.3.5 – A1.3.7 — только для HL.

⚠️ Официальных экзамен-вопросов от преподавателя по этой теме пока нет. Вопросы в разделе ниже — мои разработанные, в стиле IB-экзамена с детальными mark schemes. Используйте их для тренировки. Когда преподаватель опубликует официальный набор — заменю их.

1. A1.3.1 Роль операционной системы

Операционная система (OS) — это посредник (interface) между пользователем, приложениями и аппаратным обеспечением (hardware). Её задача — управлять системными ресурсами (CPU, память, диск, сеть, периферия) и предоставлять сервисы приложениям и пользователю: файловую систему, управление процессами, безопасность, графический интерфейс.

Примеры ОС: Linux, Windows, macOS, Android, iOS, BSD, Orbis (для PlayStation). У каждой свои сильные стороны — нет «лучшей» ОС, есть только наиболее подходящая под конкретную задачу.

🎭 Абстракция железа (hardware abstraction)

Абстракция (abstraction) — это сокрытие сложных деталей за простым интерфейсом. Когда ты играешь в видеоигру, ты не думаешь о том, как распределяются потоки, как выделяется память, как драйверы общаются с GPU. Ты просто играешь. Это и есть работа ОС — спрятать всю сложность.

Аналогия из жизни: руль и педали в машине. Тебе не нужно понимать, как работает двигатель внутреннего сгорания, чтобы ехать. Руль, педали и коробка передач — это интерфейс к сложной механике.

💡 Если бы каждое приложение само управляло памятью, диском и CPU, был бы хаос (chaos): программы конфликтовали бы за ресурсы, ломали друг друга, нарушали безопасность. ОС играет роль дирижёра оркестра — каждый «инструмент» (приложение) играет свою партию в нужный момент.
Вопрос 1.1. Опиши роль операционной системы. 3 балла

Что писать:

  • ОС — это интерфейс между пользователем/приложениями и аппаратным обеспечением.
  • Она управляет системными ресурсами (CPU, память, диск, сеть, периферия).
  • Предоставляет сервисы: файловую систему, управление процессами, безопасность, пользовательский интерфейс.
Вопрос 1.2. Объясни, что такое абстракция железа и зачем она нужна. 4 балла

Что писать:

  • Абстракция — сокрытие сложных деталей за простым интерфейсом.
  • Приложение не управляет напрямую RAM, CPU и драйверами — оно обращается к ОС через системные вызовы (system calls).
  • Это упрощает разработку: один и тот же код работает на разном железе.
  • Это повышает надёжность и безопасность: программа не может случайно сломать систему или другие приложения.

ОС не делает «одну вещь» — она делает десяток связанных вещей одновременно. Вот основные функции, которые надо знать на экзамене.

2.1 Что такое процесс (process)?

⚙️ Процесс — это выполняющаяся программа

Программа (program) — пассивный набор инструкций на диске. Процесс (process) — активное выполнение этих инструкций в памяти. Когда ты дважды кликаешь иконку, ОС создаёт процесс для управления выполнением программы. Из одной программы можно запустить несколько процессов.

У каждого процесса есть:

  • Память — приватная область для кода, данных и стека (call stack).
  • CPU-время — «слоты» процессорного времени, выделенные планировщиком.
  • Открытые файлы / сетевые соединения.
  • PID (Process ID) — уникальный идентификатор.
  • Приоритет — влияет на решения планировщика.
  • Регистры — текущие значения в CPU.

Состояния процесса (process states):

  • New — создаётся и загружается в память.
  • Ready — готов к выполнению, ждёт своей очереди на CPU.
  • Running — выполняется на CPU прямо сейчас.
  • Blocked / Waiting — ждёт внешнего события (например, данных с диска или ввода пользователя).
  • Terminated — завершён, ОС освобождает его ресурсы.

2.2 Поток (thread)

Поток (thread) — это самая маленькая единица выполнения, которой управляет планировщик. Внутри одного процесса может быть много потоков, и они делят память. Пример: текстовый редактор может иметь поток для отрисовки документа, поток для проверки орфографии в фоне и поток для обработки ввода с клавиатуры — всё одновременно.

2.3 Целостность системы (system integrity)

Изоляция процессов

Каждый процесс работает в своей виртуальной памяти — не может случайно сломать другой процесс или саму ОС.

Защита памяти

Memory protection — отдельные адресные пространства, которые ОС не даёт перезаписать чужим процессам.

User mode vs kernel mode

Прямой доступ к железу — только в режиме ядра (kernel mode). Приложения работают в режиме пользователя (user mode).

Sandboxing

Ограничение приложения только теми ресурсами, которые ему нужны (файлы, сеть, память) — снижает риск от уязвимостей.

Права доступа (ACL)

Каждый файл/каталог имеет права read/write/execute для владельца, группы и других. Это контролирует, кто и что может делать.

CIA-триада

Конфиденциальность (Confidentiality), Целостность (Integrity), Доступность (Availability) — рамка для оценки безопасности.

Шифрование

BitLocker (Windows), FileVault (macOS), dm-crypt/LUKS (Linux) — шифруют диск, защищая данные при краже.

Firewall + антивирус

Firewall фильтрует пакеты на сетевом уровне; антивирус ищет вредоносное поведение и сигнатуры известных вирусов.

Patch management

Обновления безопасности, исправления багов, новые функции. Часто проверяют целостность и подлинность обновления.

Rollback (откат)

Возврат к последнему рабочему состоянию: Windows System Restore, macOS Time Machine.

Контрольные суммы / хеши

Checksums и hashes проверяют целостность файлов — изменилось ли что-то по сравнению с эталоном.

Мониторинг и логирование

Monitoring — текущее состояние. Logging — журнал событий (Event Viewer в Windows, Console в macOS, /var/log в Linux).

2.4 Управление памятью (memory management)

ОС выделяет и освобождает память процессам, защищает адресные пространства друг от друга, и реализует виртуальную память (virtual memory). Виртуальная память позволяет процессам «думать», что у них больше памяти, чем есть физически — за счёт пейджинга (paging): лишние страницы памяти временно перекладываются на диск (swap space).

Page (страница) — это фиксированный непрерывный блок виртуальной памяти. Без виртуальной памяти такие блоки называются page frames. MMU (Memory Management Unit) в железе помогает ОС динамически отображать виртуальные адреса на физические.

2.5 Файловая система (file system)

Метод, которым ОС хранит и читает данные. Организует данные блоками на диске, определяет правила именования файлов, структуру каталогов, права доступа. Примеры: NTFS (Windows), ext4 (Linux), APFS (macOS).

2.6 Управление устройствами (device management)

ОС общается с аппаратурой через драйверы (drivers). Драйвер переводит команды ОС в специфические для устройства сигналы. Когда ты двигаешь мышь, она генерирует прерывание — драйвер передаёт его в ОС, та обновляет позицию курсора. Драйверы также дают единый интерфейс приложениям: они не должны знать, какая конкретно мышь подключена.

2.7 Планирование (scheduling)

Решение, кто и когда получает время CPU. Учитывает приоритет, справедливость, эффективность, нагрузку. Подробно — в разделе A1.3.3.

2.8 Безопасность (security)

ОС предоставляет firewall, шифрование, контроль доступа, аутентификацию пользователя, патчи безопасности, sandboxing, логирование, проверки целостности данных.

2.9 Учёт ресурсов (accounting)

ОС отслеживает использование ресурсов: сколько CPU-времени, памяти, диска, сетевого трафика «съел» каждый процесс. Раньше это нужно было, чтобы выставлять счета пользователям мейнфреймов. Сегодня — снова актуально для облачных вычислений (cloud computing), где платишь «за использование».

2.10 Графический интерфейс (GUI)

Windows, macOS и большинство Linux-сред имеют разные GUI, но их роль одна: визуальный интерфейс с иконками, окнами, меню, кнопками. Хороший GUI упрощает использование; плохой GUI мешает работе. Эта тема — часть human–computer interaction (HCI).

2.11 Виртуализация (virtualization)

🏢 Один физический сервер, много виртуальных машин

Виртуализация — создание виртуальных копий физического железа. Один компьютер может «хостить» несколько виртуальных машин (VM), и каждая работает так, будто у неё своё собственное железо. Аналогия: один большой дом разделили на отдельные квартиры — каждая со своими «комнатами» (ресурсами), но в одном здании.

Ключевая технология — гипервизор (hypervisor): программный слой между виртуальными машинами и физическим железом. Гипервизор распределяет ресурсы между VM и изолирует их друг от друга.

Два главных преимущества:

  • Эффективное использование железа — несколько VM делят один CPU/память/диск, повышая загрузку.
  • Изоляция — сбой или взлом одной VM не влияет на другие.

2.12 Сетевые функции (networking)

ОС реализует сетевой стек (network stack) — слои TCP/IP. Управляет сетевыми картами через драйверы, настраивает IP-адреса (статически или через DHCP), применяет шифрование (SSL/TLS), фильтрует пакеты файрволом. В комплекте идут утилиты: ping, ifconfig/ipconfig, netstat, traceroute/tracert.

Вопрос 2.1. Опиши, как операционные системы обрабатывают хранилище и файловые системы. 3 балла

Что писать:

  • ОС использует файловую систему как слой абстракции над физическим диском (HDD/SSD).
  • Файловая система определяет, как данные именуются, организуются в каталоги, хранятся в блоках на диске.
  • ОС управляет правами доступа, целостностью файлов и выделением свободного места (например, NTFS в Windows, ext4 в Linux, APFS в macOS).
Вопрос 2.2. Опиши, как операционные системы обеспечивают целостность системы и управляют фоновыми процессами. 3 балла

Что писать:

  • ОС использует изоляцию процессов и защиту памяти, чтобы один процесс не мог сломать другой или саму ОС.
  • Фоновые процессы (background processes) работают без видимого интерфейса; ОС управляет их приоритетами и доступом к ресурсам.
  • Логирование, контрольные суммы, патчи и rollback поддерживают доверенное состояние системы.
Вопрос 2.3. Опиши важность сетевых возможностей в операционных системах, включая управление сетевым железом и протоколами. 4 балла

Что писать:

  • ОС реализует сетевой стек (TCP/IP) — собирает входящие пакеты обратно в данные для приложений.
  • Управляет сетевыми картами (NIC) через драйверы, абстрагируя различия между разным железом.
  • Назначает IP-адреса статически или через DHCP, что необходимо для идентификации устройства в сети.
  • Применяет безопасность: firewall, SSL/TLS-шифрование, аутентификацию; даёт диагностические утилиты ping, traceroute, netstat.

3. A1.3.3 Подходы к планированию

Планирование (scheduling) — это решение, в каком порядке и сколько времени каждому процессу даётся на CPU. ОС оценивает эффективность планировщика через throughput (сколько процессов завершено за единицу времени), turnaround time (сколько прошло от поступления до завершения), waiting time и CPU utilization.

3.1 Ключевые термины

Slice (квант времени, срез) — сколько CPU-времени даётся процессу, прежде чем его прервут или заменят.
Quantum — фиксированный квант времени в pre-emptive системах (например, round robin).
Pre-emption (вытеснение) — способность планировщика прервать выполняющийся процесс и отдать CPU более приоритетному.
Fairness (справедливость) — равноправное распределение CPU между всеми процессами.
Starvation (голодание) — процесс никогда не получает CPU, потому что другие постоянно мешают.
Efficiency (эффективность) — максимум throughput, минимум waiting time.
Context switching (переключение контекста) — сохранение состояния одного процесса и загрузка другого; занимает время и снижает производительность.
Convoy effect (эффект каравана) — короткие процессы ждут одного длинного, всё замедляется.
Ageing (старение) — приоритет ожидающего процесса постепенно повышают, чтобы избежать starvation.

3.2 Четыре алгоритма планирования

🚶 FCFS — First-Come, First-Served

Процессы получают CPU в том порядке, в котором пришли. CPU обслуживает один процесс до конца, потом следующий. Не вытесняющий (non-pre-emptive): начатый процесс работает до завершения.

Аналогия: очередь в кассе — кто раньше подошёл, тот раньше обслужен.

Где использовать: простые среды с предсказуемой длиной задач, когда важна простота и предсказуемость, а не максимальная пропускная способность.

🔁 Round Robin

Вытесняющий (pre-emptive) алгоритм. Каждому процессу даётся фиксированный квант времени (time quantum). Если процесс не завершился — его отправляют в конец очереди, следующий получает CPU. Круг повторяется — отсюда «round robin».

Аналогия: учитель опрашивает учеников по очереди, каждому даёт 30 секунд.

Тонкость: размер кванта критичен. Слишком короткий — много переключений контекста (большие накладные расходы). Слишком длинный — алгоритм вырождается в FCFS.

Где использовать: системы с разделением времени (time-sharing), многозадачные системы, где важна отзывчивость для всех пользователей.

📚 Multilevel Queue Scheduling

Очередь готовых процессов разделена на несколько очередей, каждая со своим алгоритмом и приоритетом. Процессы навсегда закрепляются за одной очередью по своему типу (например, системные vs пользовательские, фоновые vs интерактивные).

Между очередями планирование может быть фиксированно-приоритетным (высокая очередь всегда первая) или по round robin между очередями.

Аналогия: в больнице — три очереди: критические пациенты, срочные, плановые. Критических обслуживают первыми, но плановые тоже не забывают.

Где использовать: сложные системы со смешанными типами процессов — например, ОС, которая балансирует системные и пользовательские задачи.

⭐ Priority Scheduling

У каждого процесса есть приоритет, и CPU получает тот, у кого приоритет выше. Бывает вытесняющим (pre-emptive) — пришёл более важный, вытеснил текущий — и невытесняющим (non-pre-emptive) — текущий доработает, потом проверим очередь.

Аналогия: в аэропорту — пассажиров с золотой картой пускают вперёд, остальные ждут.

Минус: риск starvation — низкоприоритетный процесс может «никогда» не получить CPU. Решение — ageing (постепенно повышать приоритет ожидающих).

Где использовать: системы реального времени (real-time), где критично выполнить высокоприоритетные задачи вовремя — медицинская техника, автоматика, авиатехника.

Критерий FCFS Round Robin Multilevel Priority
Справедливость Справедлив «по очереди прихода». Но convoy effect — короткие ждут долгих. Спроектирован как справедливый: каждому равный квант. По сути несправедлив: одни очереди важнее других. Риск starvation для низкоприоритетных, если нет ageing.

Pre-emption Нет (non-pre-emptive). Да (по таймеру). Зависит от алгоритма в каждой очереди. Бывает оба варианта.

Вопрос 3.1. Опиши алгоритм round robin и обсуди его потенциальные недостатки. 6 баллов

Описание (3 балла):

  • Round robin — вытесняющий алгоритм планирования.
  • Каждому процессу даётся фиксированный квант времени (time quantum).
  • Если процесс не завершился — он отправляется в конец очереди, следующий получает CPU. Цикл продолжается.

Недостатки (3 балла):

  • Выбор кванта критичен: слишком короткий → много переключений контекста, накладные расходы; слишком длинный → поведение как у FCFS.
  • Накладные расходы на context switching снижают эффективность CPU.
  • Не учитывает приоритет: срочные задачи получают столько же CPU, сколько и фоновые — плохо для real-time систем.
Вопрос 3.2. Сравни и противопоставь эффективность и справедливость round robin и priority scheduling. 6 баллов

Что писать:

  • Справедливость: round robin справедлив по построению — каждый получает равный квант. Priority — несправедлив: высокоприоритетные подавляют низкоприоритетные, риск starvation.
  • Эффективность: priority может быть очень эффективен, если приоритеты заданы правильно — критичные задачи выполняются вовремя. Round robin даёт среднее время отклика, но платит overhead за частые переключения.
  • Применение: round robin лучше для time-sharing, многопользовательских систем; priority — для real-time, медицины, авиации.
  • Smягчение слабостей: у priority — ageing (постепенно повышать приоритет ожидающих); у round robin — подбор оптимального кванта.
Вопрос 3.3. Объясни, как multilevel queue scheduling можно использовать для управления процессами разных типов и приоритетов. 4 балла

Что писать:

  • Multilevel queue делит готовые процессы на несколько очередей по типу или приоритету (например, системные/пользовательские, foreground/background).
  • Каждая очередь имеет свой алгоритм внутри (одна — FCFS, другая — round robin).
  • Между очередями — фиксированный приоритет или round robin.
  • Преимущество: можно эффективно обработать смешанную нагрузку — критичные системные задачи получают приоритет, но и пользовательские не голодают.

4. A1.3.4 Polling vs Interrupts

Два способа, которыми CPU узнаёт, что внешнее устройство или программа требует внимания.

🔔 Interrupt (прерывание)

Устройство само посылает сигнал процессору: «эй, у меня данные, обработай!». CPU прерывает текущую работу, сохраняет контекст и запускает Interrupt Service Routine (ISR) — специальную процедуру для этого события. После ISR процессор возвращается к прерванному делу.

Аналогия: ты работаешь, и звонит дверной звонок. Ты прерываешься, идёшь открыть, потом возвращаешься к работе.

🔄 Polling (опрос)

Процессор сам периодически опрашивает устройства: «есть ли что-то новое?». Если нет — возвращается к работе и через какое-то время опросит снова. Если да — обрабатывает.

Аналогия: ты ждёшь курьера, и каждые 5 минут идёшь к двери проверить, не пришёл ли он.

4.1 Сравнительная таблица факторов (Таблица 19 в учебнике)

Фактор Interrupt Polling
Частота событий Лучше при частых событиях: CPU реагирует сразу, минимизируя ожидание. Подходит для редких событий: CPU может позволить себе периодически проверять.
Накладные расходы CPU Overhead от сохранения/восстановления контекста и выполнения ISR. Overhead от пустых проверок — CPU тратит циклы, даже если ничего не происходит.
Источник питания Лучше для батарей: CPU может уйти в low-power sleep между прерываниями. Высасывает батарею — CPU всё время должен быть «бодрым» для опросов.
Предсказуемость событий Лучше для непредсказуемых событий — реагирует когда угодно. Эффективен при регулярных, известных интервалах.
Контролируемая задержка (latency) Низкая и контролируемая latency — реакция как только событие произошло. Переменная latency — зависит, когда событие произошло относительно цикла опроса.
Безопасность Уязвимы для DoS-атак: поток прерываний может перегрузить CPU. Меньше подвержен таким атакам, но может пропустить аномалию, если её не ожидает.

4.2 Реальные сценарии

Клавиатура и мышь

Используют прерывания — для мгновенной реакции. Polling дал бы заметные задержки. Игровые мыши работают на 1000 Hz polling rate для минимальной задержки.

Сетевые коммуникации

Пакеты приходят непредсказуемо — нужны прерывания. NIC «будит» CPU, как только пришёл пакет.

Дисковые I/O

Диски медленные по сравнению с CPU — прерывания позволяют CPU делать другое, пока диск работает, и сообщают, когда чтение/запись готовы.

Встроенные системы (IoT)

Прерывания критичны для экономии батареи: датчик «спит», просыпается только при событии.

Real-time системы

Медицинская техника, автомобильные контроллеры — прерывания дают гарантированную low latency.

Когда подходит polling

Если события предсказуемы, редки, а устройство работает от сети (mains) — polling проще в реализации.

4.3 Пример: что происходит, когда ты двигаешь мышь

  1. Физическое движение: ты двигаешь мышь, её датчики ловят движение.
  2. Генерация сигнала: сенсор превращает движение в электрические сигналы.
  3. Формирование пакета данных: внутренний процессор мыши собирает данные о направлении и кнопках.
  4. Передача: данные идут на компьютер по USB, PS/2 или беспроводно (Bluetooth, RF).
  5. Сигнал прерывания: USB- или Bluetooth-интерфейс посылает interrupt процессору.
  6. Активация ISR: CPU сохраняет состояние, запускает Interrupt Service Routine для мыши.
  7. Расчёт позиции курсора: ОС вычисляет новую позицию с учётом «скорости указателя».
  8. Обновление экрана: ОС шлёт GPU команду перерисовать курсор.
  9. Возврат: CPU восстанавливает состояние и возвращается к прерванной задаче.
Вопрос 4.1. Объясни метод polling и обсуди его потенциальные недостатки по части эффективности CPU. 6 баллов

Описание (3 балла):

  • Polling — процессор периодически опрашивает устройства, чтобы узнать, есть ли у них данные или нужно ли их обслуживать.
  • Это цикл проверок состояния всех устройств по очереди.
  • Polling прост в реализации.

Недостатки (3 балла):

  • Тратит CPU-циклы на пустые проверки, когда событий нет → снижает эффективность.
  • Вводит переменную задержку: реакция зависит от момента опроса.
  • Высасывает батарею — CPU не может уйти в спящий режим.
Вопрос 4.2. Сравни и противопоставь interrupts и polling по части энергопотребления и пригодности для устройств на батарее. 6 баллов

Что писать:

  • Interrupts позволяют CPU уходить в low-power sleep mode и просыпаться только при событии — это сильно экономит батарею.
  • Polling требует, чтобы CPU был активен и периодически опрашивал устройства, даже если ничего не происходит — батарея садится быстрее.
  • На устройствах с питанием от сети (mains) разница менее критична.
  • Вывод: для мобильных устройств, носимой электроники, IoT и встроенных систем interrupts предпочтительнее; для стационарных систем выбор более свободный.
Вопрос 4.3. Объясни роль controlled latency при выборе между interrupt handling и polling для real-time систем. 6 баллов

Что писать:

  • Controlled latency — способность гарантировать предсказуемую задержку между событием и реакцией системы.
  • В real-time системах (медицина, автомобили, промышленность) реакция должна укладываться в строгие временные рамки — иначе риск аварии.
  • Interrupts дают низкую и контролируемую latency — CPU реагирует, как только пришёл сигнал.
  • Polling даёт переменную latency — реакция зависит от того, успел ли цикл опроса; в худшем случае задержка = период опроса.
  • Вывод: в real-time системах interrupts практически безальтернативны.
Вопрос 4.4. Оцени, чем отличаются проблемы безопасности при interrupt handling и polling. 6 баллов

Что писать:

  • Interrupts могут быть использованы для DoS-атаки: вредоносное устройство или программа заваливает CPU прерываниями, и легитимные задачи перестают выполняться.
  • Polling в этом смысле менее уязвим — CPU всё равно опрашивает в своём ритме.
  • Но polling может пропустить аномалию, если она происходит между опросами или не соответствует тому, что система ожидает.
  • Защита от DoS на прерываниях — ограничение частоты прерываний, проверка источника, использование прерываний с приоритетом.

5. A1.3.5 Многозадачность и распределение ресурсов HL

Многозадачность (multitasking) — способность ОС управлять и выполнять несколько процессов «одновременно». На самом деле в одно мгновение CPU выполняет один процесс, но переключается так быстро, что создаётся иллюзия параллельности. Этот механизм называется context switching (переключение контекста) — ОС сохраняет состояние текущего процесса (регистры, указатели памяти) и загружает состояние следующего.

Распределение ресурсов (resource allocation) — эффективное раздавание CPU, памяти, диска, сети между процессами. Это сложная динамичная задача — миллионы решений в секунду.

5.1 Механизмы многозадачности

🎛️ Как ОС жонглирует сотнями процессов

  • Process management: ОС ведёт таблицу процессов (process table) с состоянием (running/waiting/blocked), приоритетом, использованием ресурсов.
  • Context switching: при переключении ОС сохраняет состояние в таблице и загружает следующий — обычно за микросекунды.
  • Многопоточность (multithreading): внутри процесса несколько потоков разделяют память и могут выполняться параллельно на multi-core CPU.

5.2 Механизмы распределения ресурсов

Управление памятью

Paging — разделение физической памяти на фиксированные блоки (страницы). Segmentation — деление памяти на сегменты разного размера (код, данные). Virtual memory = paging + segmentation: процессам кажется, что памяти больше, чем физически есть, за счёт swap-файла на диске.

Управление устройствами

ОС через драйверы общается с устройствами, предотвращает конфликты, организует доступ к периферии (принтеры, сетевые карты).

Файловые системы

Абстракция над диском: NTFS, ext4, APFS. Файлы и каталоги — поверх блоков на физическом носителе.

Мониторинг и лимиты

ОС следит за тем, сколько каждый процесс ест CPU, памяти, I/O. Может выставить квоты (quotas), чтобы один процесс не съел всё.

5.3 Вызовы планирования задач (task scheduling)

  • Fairness vs efficiency: справедливо раздать CPU всем — но высокоприоритетные не должны страдать, а низкоприоритетные не должны умирать.
  • Pre-emption: приостановка процесса в пользу более важного — даёт отзывчивость, но добавляет накладные расходы.
  • Starvation: в крайних случаях низкоприоритетные задачи никогда не получают CPU — нужны стратегии вроде ageing.

5.4 Resource contention (борьба за ресурсы)

Происходит, когда несколько процессов хотят один и тот же ресурс, который не делится на всех. Это может привести к нескольким плохим состояниям:

🔒 Deadlock (взаимная блокировка)

Два или больше процессов ждут друг друга по кругу — никто не может продолжить. Классический пример: процесс A держит ресурс 1 и хочет ресурс 2; процесс B держит ресурс 2 и хочет ресурс 1. Оба зависнут навсегда.

Что делает ОС:

  • Detection (обнаружение): алгоритмы мониторят зависимости ресурсов и ловят циклы ожидания.
  • Release the deadlock: ОС может завершить один из процессов, вытеснить его задачи, или перераспределить ресурсы.
  • Prevention (предотвращение): deadlock avoidance algorithms (например, Banker's algorithm) проверяют, не создаст ли запрос на ресурс цикл — и отказывают, если да.

🔁 Livelock

Процессы постоянно запрашивают и отпускают ресурсы, но не продвигаются вперёд. В отличие от deadlock, они активны, но прогресса нет. Аналогия: два человека столкнулись в коридоре, оба отходят в одну сторону, опять блокируют друг друга, и так бесконечно.

💥 Thrashing

Возникает, когда страниц памяти не хватает, и ОС постоянно подкачивает (swap) страницы между RAM и диском. CPU тратит больше времени на пейджинг, чем на полезную работу. Решение — оптимизация demand paging и виртуальной памяти, добавление RAM.

Вопрос 5.1. Объясни механизмы, которые ОС использует для распределения ресурсов, включая управление памятью и устройствами. 6 баллов

Что писать:

  • Память: paging (фиксированные страницы), segmentation (логические сегменты), virtual memory (расширение RAM за счёт диска).
  • Устройства: ОС использует драйверы как абстракцию, предотвращает конфликты, организует доступ.
  • Мониторинг: ОС следит за использованием CPU, памяти, I/O и применяет лимиты.
  • Файловая система: блочный уровень хранения, организация в каталоги и файлы.
  • Network stack: распределение пропускной способности между процессами.
  • Цель: справедливость, эффективность, стабильность.
Вопрос 5.2. Обсуди вызовы, связанные с task scheduling, включая fairness, pre-emption и starvation. 6 баллов

Что писать:

  • Fairness: ОС должна давать CPU всем процессам, но высокоприоритетные нуждаются в большем времени — компромисс.
  • Pre-emption: возможность прервать процесс ради более срочного — улучшает отзывчивость, но context switching стоит времени.
  • Starvation: низкоприоритетный процесс может никогда не получить CPU; нужны стратегии вроде ageing (постепенное повышение приоритета ожидающих).
  • Выбор алгоритма: ни один не идеален — выбор зависит от нагрузки.
  • Динамика: в реальной системе нагрузка меняется ежесекундно, ОС адаптирует решения.
Вопрос 5.3. Обсуди проблему resource contention в многозадачных средах и стратегии её смягчения. 6 баллов

Что писать:

  • Resource contention — несколько процессов хотят один ресурс одновременно (CPU, память, файл, периферия), а он не делится.
  • Последствия: deadlock (взаимная блокировка), livelock (активное ожидание без прогресса), thrashing (избыточный пейджинг).
  • Стратегии: алгоритмы обнаружения deadlock, избегания (Banker's algorithm), корректное использование блокировок (locks) и семафоров.
  • Восстановление: завершение одного из процессов, освобождение ресурсов, откат.
  • Профилактика: заказывать ресурсы в одном и том же порядке, использовать timeouts на ожидание.
Вопрос 5.4. (a) Очерти важность управления deadlock в операционных системах. 2 балла (b) Опиши, как ОС предотвращает и разрешает deadlock. 4 балла

(a) Важность (2 балла):

  • Без управления deadlock система может зависнуть полностью, потому что процессы вечно ждут друг друга.
  • Это критично для надёжности, особенно в серверах, базах данных, real-time системах.

(b) Предотвращение и разрешение (4 балла):

  • Detection: ОС анализирует граф зависимостей ресурсов и ищет циклы.
  • Release: при обнаружении — завершить один из процессов, вытеснить задачи, перераспределить ресурсы.
  • Prevention: deadlock avoidance algorithms (например, Banker's algorithm) отказывают в запросе ресурса, если это создаст цикл.
  • Дизайн: требовать процессы заказывать ресурсы в едином порядке — исключает один из условий deadlock.

6. A1.3.6 Компоненты управляющей системы HL

Управляющая система (control system) — это устройство или набор устройств, который управляет, направляет или регулирует поведение других устройств через петли управления (control loops). Примеры: лифт, стиральная машина, светофор, термостат, круиз-контроль.

6.1 Модель IPO + обратная связь

IPO (Input – Process – Output) — базовая модель: данные поступают на вход → обрабатываются → дают выход. Feedback (обратная связь) — выход возвращается обратно в систему, чтобы скорректировать дальнейшую работу.

ЭлементОпределениеПример (круиз-контроль)
Input Данные / сигналы, поступающие в систему: команды пользователя, данные с датчиков. Водитель ставит желаемую скорость 100 км/ч; датчик измеряет текущую скорость.
Process Обработка входных данных в контроллере или микроконтроллере по предзаданным правилам или алгоритму. Контроллер сравнивает желаемую и текущую скорость, считает разницу (error), решает, насколько прибавить газу.
Output Реальное действие / поведение системы: движение, нагрев, открытие клапана. Актуатор дроссельной заслонки увеличивает подачу топлива — машина разгоняется.
Feedback Выход замеряется датчиком и возвращается в систему — для корректировки в реальном времени. Датчик скорости постоянно мерит реальную скорость — это closed-loop система.

6.2 Open-loop vs Closed-loop

Сверху: open-loop. Снизу: closed-loop с обратной связью от датчика к контроллеру.

Open-loop (без обратной связи)

Выполняет предзаданный набор операций и не корректируется по результату. Пример: обычный тостер — таймер на 2 минуты, неважно, поджарился хлеб или сгорел. Или сушилка для одежды на таймере.

Closed-loop (с обратной связью)

Непрерывно мониторит выход, сравнивает с целью и корректирует входы. Пример: холодильник Samsung работает по closed-loop: датчик температуры → контроллер → компрессор включается/выключается. Также: круиз-контроль, термостат, автопилот.

6.3 Пять ключевых компонентов (Таблица 21 в учебнике)

🧠 Controller (контроллер)

Устройство или ПО, которое обрабатывает входы от датчиков и посылает сигналы актуаторам. Может быть встроенным микроконтроллером или промышленным PLC (Programmable Logic Controller).

Пример: На сборочной линии PLC координирует последовательность машин — мотор крутит конвейер, пневматическая рука переставляет деталь, датчик проверяет результат.

🌡️ Sensors (датчики)

Устройства, которые обнаруживают и измеряют физические величины (температура, свет, давление, движение, влажность) и превращают их в сигналы, понятные контроллеру.

Пример: Thermocouple (термопара) в промышленной печи — мерит температуру и шлёт контроллеру, который регулирует подачу топлива.

⚙️ Actuators (актуаторы / исполнительные устройства)

Механические или электронные устройства, которые совершают физическое действие в ответ на сигналы контроллера: моторы, клапаны, нагреватели, сирены.

Пример: Solenoid valve (соленоидный клапан) в автоматической поливе — открывает/закрывает воду по команде контроллера, который читал данные с датчика влажности почвы.

🔄 Transducer (преобразователь)

Превращает одну форму энергии или сигнала в другую. Часто стоит между физическим миром и электроникой. Технически датчики и актуаторы — частный случай преобразователей.

Пример: Piezoelectric pressure sensor в системе мониторинга давления шин — преобразует давление в электрический сигнал, который читает бортовой компьютер; если давление вне нормы, водителю показывают предупреждение.

📐 Control algorithm (алгоритм управления)

Математическая формула или логика, которой контроллер пользуется, чтобы решить, какой выход дать на основе входа, цели и текущего состояния.

Пример: PID-контроль (proportional–integral–derivative) в домашнем отоплении — мерит ошибку (разницу между целевой и фактической температурой), её интеграл (накопленную ошибку) и производную (скорость изменения), сглаживает регулирование. Меньше скачков, точнее температура, меньше расход энергии.

Вопрос 6.1. Назови четыре главных компонента в управляющих системах. 4 балла

Что писать (по 1 баллу за каждый):

  • Input — данные от пользователя или датчиков.
  • Process — обработка данных контроллером.
  • Output — действие, выполняемое системой (через актуатор).
  • Feedback — обратная связь от выхода обратно в систему для коррекции.

Альтернативный набор «компонентов» — controller, sensors, actuators, control algorithm.

Вопрос 6.2. Очерти роль feedback в управляющей системе. 2 балла

Что писать:

  • Feedback возвращает информацию о выходе обратно в систему.
  • Это позволяет системе корректировать поведение в реальном времени, чтобы удерживать желаемое состояние (closed-loop control).
Вопрос 6.3. Объясни функции датчиков и актуаторов в управляющей системе, включая как они взаимодействуют с другими компонентами. 6 баллов

Что писать:

  • Датчики (sensors) измеряют физические величины (температура, давление, движение) и превращают их в электрические сигналы для контроллера.
  • Контроллер обрабатывает эти сигналы через control algorithm.
  • Актуаторы (actuators) получают сигналы от контроллера и выполняют физическое действие (открыть клапан, запустить мотор).
  • В closed-loop системе датчики затем мерят результат и шлют его обратно — петля замыкается.
  • Преобразователи (transducers) часто стоят в роли датчика или актуатора, переводя сигнал из одной формы в другую.
  • Пример: в холодильнике термопара (датчик) мерит температуру, контроллер сравнивает с целью, реле (актуатор) запускает компрессор.

7. A1.3.7 Реальные применения управляющих систем HL

Управляющие системы повсюду. Все они строятся на одной модели: датчики собирают данные → контроллер с алгоритмом принимает решение → актуаторы выполняют действие → датчики снова мерят результат. Меняется только сложность алгоритма и набор сенсоров.

🚗 Автономные транспортные средства (autonomous vehicles)

Сложнейшие управляющие системы.

  • Датчики: LIDAR (3D-картирование, обнаружение объектов), radar (плохая погода), камеры (визуальное распознавание), ultrasonic (ближняя проксимити), IMU (отслеживание движения самой машины).
  • Алгоритмы: path planning (планирование маршрута), localization (где я нахожусь), obstacle avoidance (избегание препятствий), AI для интерпретации трафика.
  • Актуаторы: электромеханический руль, electronic throttle, ABS (anti-lock braking).
  • Реальное: Tesla Autopilot, Waymo. Мониторинг 360° на скорости 100+ км/ч в реальном времени.

🌡️ Домашние термостаты и BMS (building management systems)

  • Датчики: биметаллические полоски или цифровые термисторы.
  • Контроллер: простое on/off или сложный PID-алгоритм.
  • Актуаторы: реле или переключатели для печки/кондиционера.
  • BMS (масштаб здания): сеть температурных датчиков, центральный контроллер учитывает занятость, погоду, цену энергии; продвинутые BMS используют ML для оптимизации.
  • Реальное: Nest Thermostat — учится привычкам и экономит до 15% энергии.

🛗 Автоматическое управление лифтами

  • Цель: минимизировать время ожидания и поездки пассажиров.
  • Датчики: датчики положения каждой кабины, кнопки вызова на этажах и внутри.
  • Контроллер: алгоритм назначает кабину на вызов — учитывает направление движения, нагрузку, расстояние.
  • Актуаторы: моторы перемещают кабину, двери.
  • Реальное: небоскрёб с 10 лифтами использует destination dispatch — пассажир заранее указывает этаж, система группирует попутных.

🧺 Автоматические стиральные машины

  • Датчики: датчики веса белья, уровня воды, turbidity sensor (мутность воды — определяет грязь).
  • Контроллер: fuzzy logic или классические алгоритмы — выбирает длительность цикла, температуру, скорость вращения.
  • Актуаторы: входной клапан воды, слив, мотор барабана, нагреватель.
  • Реальное: современные машины подстраивают цикл — мало белья → меньше воды и времени; очень грязное → дольше стирка.

🚦 Управление светофорами

  • Датчики: inductive loop detectors в дорожном полотне (магнитное поле меняется от металла машины), камеры с computer vision, кнопки для пешеходов.
  • Контроллер: старые — pre-timed schedule (фиксированный цикл); современные — actuated controllers с динамическим таймингом; продвинутые ATCS используют ИИ по всему городу.
  • Актуаторы: сами лампы светофора (красный, жёлтый, зелёный) и пешеходные сигналы.
  • Реальное: Москва, Сингапур, многие крупные города используют AI-управляемые светофоры — снижают пробки на 20–30%.

💧 Управление поливом (irrigation control systems)

  • Датчики: tensiometers (натяжение воды в почве), capacitance-based sensors (прямое измерение влажности).
  • Контроллер: программы под конкретную культуру, иногда интегрируются с прогнозом погоды.
  • Актуаторы: соленоидные клапаны на линиях полива.
  • Реальное: variable-rate irrigation — разные зоны поля получают разную воду по локальным датчикам. Экономия до 30% воды.

🚨 Домашние охранные системы (home security)

  • Датчики: датчики движения (passive infrared, PIR), контакты на дверях и окнах, glass break detectors (акустика + датчик вибрации), датчики дыма и угарного газа.
  • Контроллер: центральная панель с логикой — отличает реальную угрозу от ложной тревоги.
  • Актуаторы: сирены, smart locks, отправка уведомлений на телефон или в охранную фирму.
  • Реальное: Ring, Arlo — сеть камер и сенсоров с уведомлениями в мобильное приложение.

🚪 Автоматические двери

  • Датчики: microwave (импульсы и анализ отражения) или infrared (тепло), optical sensors (сетка инфракрасных лучей).
  • Контроллер: логика против ложных открытий — открывать только когда подходят с определённой стороны.
  • Актуаторы: электромоторы с шестернями или гидравлика.
  • Доп. защита: light curtains — не дают двери закрыться, если кто-то в проёме.
  • Реальное: двери в супермаркетах, аэропортах, метро.
Вопрос 7.1. Назови два реальных применения управляющих систем. 2 балла

Любые два из (по 1 баллу):

  • Автономные транспортные средства.
  • Домашние термостаты / системы управления зданием (BMS).
  • Автоматические лифты.
  • Стиральные машины.
  • Светофоры.
  • Системы автоматического полива.
  • Домашние охранные системы.
  • Автоматические двери.
Вопрос 7.2. Очерти, как управляющие системы используются в автоматических стиральных машинах. 2 балла

Что писать:

  • Датчики веса белья, уровня воды и иногда мутности воды передают данные контроллеру.
  • Контроллер (часто fuzzy logic) выбирает оптимальную программу: длительность, температуру, скорость отжима, количество воды.
  • Актуаторы — клапаны воды, мотор барабана, нагреватель — выполняют программу.
Вопрос 7.3. Объясни роль датчиков, контроллеров и актуаторов в работе автономных транспортных средств. 6 баллов

Что писать:

  • Датчики: LIDAR (3D-картирование), радар (плохая погода), камеры (распознавание), ультразвук (ближняя проксимити), IMU (движение машины) — собирают полную картину окружения.
  • Контроллер: запускает алгоритмы path planning, localization, obstacle avoidance, иногда ИИ для интерпретации трафика и принятия решений.
  • Актуаторы: электромеханический руль, электронный газ, ABS — физически реализуют решения системы.
  • Closed-loop: датчики постоянно мерят результат, машина корректирует курс в реальном времени.
  • Безопасность: избыточность сенсоров (если один откажет — другие подстрахуют).
  • Сложность: миллионы решений в секунду на скоростях, опасных для человека.
Вопрос 7.4. Объясни, как системы управления светофорами используют датчики и контроллеры для регулирования трафика на перекрёстках. 6 баллов

Что писать:

  • Датчики: inductive loop detectors в полотне дороги ловят машины через изменение магнитного поля; камеры с computer vision считают и классифицируют машины; pressure-sensitive pads для пешеходов.
  • Контроллер: обрабатывает данные сенсоров. Старые работают по фиксированному расписанию; actuated controllers подстраивают тайминги под реальный поток; ATCS (adaptive traffic control systems) используют ИИ по сети перекрёстков.
  • Актуаторы: сами огни светофора (красный, жёлтый, зелёный) и пешеходные сигналы.
  • Closed-loop: датчики мерят результат (поток уменьшился? пробка ушла?) — контроллер подстраивается.
  • Польза: меньше пробок, меньше выбросов, безопаснее для пешеходов.
Вопрос 7.5. Оцени использование управляющих систем в домашней охране, акцент на типах датчиков и их функциях. 6 баллов

Что писать:

  • Датчики движения PIR — ловят инфракрасное излучение тела; экономичны, но возможны ложные срабатывания.
  • Магнитные контакты на дверях и окнах — сработают на размыкание; просты и надёжны.
  • Glass break detectors — акустика плюс датчик вибрации; ловят именно звук бьющегося стекла.
  • Датчики дыма / угарного газа — защита от пожаров и отравления, не только от взлома.
  • Контроллер сравнивает сигналы с логикой угрозы — снижает ложные тревоги.
  • Актуаторы: сирены, smart locks, push-уведомления — реакция должна быть быстрой и заметной.
  • Оценка: комбинация датчиков даёт redundancy — лучшая защита, но дороже; ложные срабатывания — главная проблема, и хороший контроллер их фильтрует.

🏠 Домашнее задание: Разбери одну управляющую систему дома

Цель: Перевести модель IPO в реальное устройство.

  1. Выбери одно устройство дома: микроволновка, посудомойка, кондиционер, духовка, робот-пылесос, увлажнитель.
  2. Опиши каждый компонент: input, process, output, feedback. Это open-loop или closed-loop? Обоснуй.
  3. Назови конкретные датчики, контроллер, актуаторы, control algorithm (если знаешь — какой алгоритм).
  4. Что произойдёт, если отключить feedback? Опиши на примере.
  5. Подумай о безопасности: что если контроллер сломается, а актуатор продолжит работать?

Что важно запомнить

  • ОС — посредник между пользователем/приложениями и железом; её главная сила — абстракция (abstraction).
  • Десять функций ОС: память, файлы, устройства, сеть, UI, безопасность, процессы, питание, периферия, планирование. Можешь назвать любые 4–6 — этого достаточно для большинства вопросов.
  • Процесс ≠ программа: программа — пассивный код, процесс — активное выполнение.
  • Планирование: FCFS (просто), round robin (справедливо), multilevel (смешано), priority (real-time). На экзамене различай по двум осям — fairness vs efficiency и pre-emptive vs non-pre-emptive.
  • Polling vs Interrupts: interrupts для частых, непредсказуемых, low-latency и batteries; polling для редких, регулярных, mains.
  • HL — многозадачность: ОС жонглирует процессами через context switching. Главные риски — deadlock, starvation, thrashing; на каждое есть стратегия (detection/avoidance, ageing, оптимизация памяти).
  • HL — управляющая система: IPO + feedback. Open-loop — без обратной связи (тостер на таймере), closed-loop — с обратной связью (термостат, круиз-контроль).
  • HL — пять компонентов: controller, sensors, actuators, transducer, control algorithm. Запомни конкретные примеры: PLC, thermocouple, solenoid valve, piezoelectric sensor, PID.
  • HL — реальные применения: 8 канонических примеров — авто-машины, термостаты, лифты, стиралки, светофоры, полив, охрана, двери. Для каждого знай датчик → контроллер → актуатор.
  • На экзамене часто спрашивают «сравни X и Y» — отвечай через таблицу: fairness, efficiency, pre-emption, use case для планирования; event frequency, CPU overhead, power, latency, security для polling vs interrupts.
