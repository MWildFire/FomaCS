Трансляция HL

Эта тема (A1.4 — Translation, только HL) — про то, как код, который ты пишешь на Python, Java или C++, превращается в машинные инструкции, которые понимает процессор. Есть три основных подхода: компилятор (compiler) переводит всю программу заранее, интерпретатор (interpreter) — строка за строкой во время выполнения, а JIT (just-in-time) — гибрид, который компилирует «на лету». Каждый подход — это компромисс между скоростью разработки, скоростью выполнения, портируемостью и моментом обнаружения ошибок.

1. Что такое трансляция и зачем она нужна

Процессор (CPU) понимает только машинный код (machine code) — последовательности двоичных бит, каждая из которых кодирует одну простую операцию: «загрузи число в регистр», «сложи два числа», «прыгни на адрес». Эти команды очень близки к железу и зависят от архитектуры (x86, ARM, MIPS — у каждой свой набор инструкций).

Человек писать в машинных кодах не может: они нечитаемы, привязаны к конкретному процессору и громоздки. Поэтому программисты пишут на языках высокого уровня (Python, Java, C++, JavaScript, Rust, Go и т.д.), а специальные программы — трансляторы — переводят этот код в машинные инструкции. Трансляция (translation) — это и есть процесс превращения человеко-понятного кода в команды, которые исполняет CPU.

Уровни кода — от человека к процессору

Уровень Пример Кто читает
Высокоуровневый язык score = 42 Человек
Ассемблер (Assembly) MOV R2, #42 Человек (с трудом) и ассемблер
Машинный код (Machine code) 0110 0000 0100 0010 Только процессор
💡 Ассемблер (assembly language) — это «чуть более читаемый» машинный код. Каждая ассемблерная команда (MOV, ADD, JMP) почти один-к-одному превращается в одну машинную инструкцию. Программист может пользоваться метками вместо адресов памяти, но всё ещё работает на уровне регистров CPU. Ассемблер (assembler) как программа — это переводчик из ассемблерного кода в машинный.

2. Кратко: компилятор vs интерпретатор vs JIT

Критерий Компилятор Интерпретатор JIT-компилятор
Когда переводит Один раз, заранее (до запуска) Строка за строкой, во время выполнения Во время выполнения, «горячие» участки кода
Результат Бинарный исполняемый файл (.exe / a.out) Файла нет, читает исходник напрямую Машинный код в памяти, кэшируется
Скорость выполнения Очень быстрая (нативный машинный код) Медленная (трансляция на каждый запуск) Средняя/быстрая (после прогрева — почти как у компилятора)
Обнаружение ошибок Многие ошибки видны на этапе компиляции (до запуска) Ошибки всплывают во время выполнения, в момент исполнения строки Гибрид: часть ошибок на этапе компиляции в байткод, часть во время JIT
Портируемость Низкая: бинарь привязан к ОС и архитектуре CPU Высокая: достаточно поставить интерпретатор на любую ОС Высокая на уровне байткода; машинный код — для текущей платформы
Скорость разработки Медленнее (нужен цикл «правка → компиляция → запуск») Быстрая (правка → сразу запуск) Быстрая (правка → запуск, JIT прозрачен)
Типичные языки C, C++, Rust, Go Python, JavaScript, Ruby, PHP Java, C#, JavaScript (V8), Python (PyPy)
Где применяют Игры, ОС, системное ПО, HFT-трейдинг Скрипты, web-бэкенд, прототипы, наука Серверные приложения, мобильные, кроссплатформенный enterprise-софт
💡 Аналогия: представь, что нужно перевести книгу с английского на русский. Компилятор = переводчик переводит всю книгу заранее, ты потом читаешь готовый русский том. Интерпретатор = синхронист переводит фразу за фразой, пока ты слушаешь оратора. JIT = синхронист, который запоминает часто повторяющиеся фразы и держит их в кармане в готовом виде.

3. Компилятор (Compiler)

⚙️ A. Что делает компилятор

Определение
Компилятор переводит исходный код в машинный код один раз, до запуска программы. Результат — бинарный исполняемый файл (binary executable), который ОС загружает в память и CPU исполняет напрямую.
Этапы
Лексический анализ → синтаксический → семантический → оптимизация → генерация кода. (См. подробно ниже в разделе 6.)
Что хорошо
  • Скорость выполнения — машинный код выполняется напрямую CPU без накладных расходов на трансляцию.
  • Ошибки ловятся заранее: синтаксические, типовые, многие семантические — компилятор не создаст исполняемый файл, пока их не исправишь.
  • Оптимизация всей программы — компилятор видит весь код и может переставлять инструкции, удалять «мёртвый» код, инлайнить функции.
  • Защита исходника — пользователю отдаётся только бинарь, исходный код не нужен.
Что неудобно
  • Цикл разработки длиннее: каждое изменение требует пересборки (может занимать секунды или минуты).
  • Низкая портируемость: бинарь, собранный под Windows x86, не запустится на macOS ARM — нужна отдельная компиляция под каждую платформу.
  • На этапе компиляции больше расходуется память (компилятор строит AST, таблицы символов и т.д.).
Примеры языков
C, C++, Rust, Go. Компиляторы: GCC, Clang/LLVM, Microsoft Visual C++ (MSVC).

Пример исходника на Rust:

fn main() {
    println!("Hello, world!");
}

После команды rustc main.rs компилятор создаёт исполняемый файл main (или main.exe на Windows), который при запуске напрямую выводит строку — без участия Rust-инструментов.

4. Интерпретатор (Interpreter)

📜 B. Что делает интерпретатор

Определение
Интерпретатор читает исходный код строка за строкой и исполняет каждую инструкцию на лету. Бинарного файла нет — программа существует только как текст исходника плюс работающий интерпретатор.
Как работает
  • Берёт следующую строку.
  • Разбирает её (лексика, синтаксис, семантика — на лету).
  • Превращает в байткод или внутреннее представление.
  • Выполняет.
  • Если встретилась ошибка — останавливается на этой строке, всё что было до — уже исполнено.
Что хорошо
  • Очень быстрый цикл разработки: правишь файл, сразу запускаешь — без шага компиляции.
  • Высокая портируемость: один и тот же скрипт работает на Windows, macOS, Linux, лишь бы стоял интерпретатор.
  • Меньше памяти под трансляцию (нет промежуточного бинаря).
  • Динамическая типизация чаще всего: меньше «бойлерплейта», быстрее писать.
  • Интерактивность: REPL (Python, JS-консоль) позволяет пробовать код прямо в командной строке.
Что неудобно
  • Медленнее выполняется: каждая строка переводится заново при каждом запуске.
  • Ошибки всплывают поздно — только когда строка реально исполнится. Опечатка в редко вызываемой функции может пройти в продакшен.
  • Нужен интерпретатор у пользователя — программу нельзя просто «отдать как .exe».
Примеры языков
Python, JavaScript, Ruby, PHP.

Пример на Python:

print("Hello, World!")

Запуск: python hello.py. Интерпретатор сам разберёт, скомпилирует в байткод и тут же выполнит. Никакого .exe на диске не появится.

💬 Реальная история (из учебника): когда команда Кэти Боуман получила первое изображение чёрной дыры, обработка огромных объёмов данных с радиотелескопов делалась на Python — потому что интерпретируемый язык позволял быстро менять алгоритм, прогонять заново и видеть результат, не теряя время на пересборку.

5. JIT-компиляция и байткод-интерпретаторы

⚡ C. JIT (Just-in-time compilation)

Определение
Гибрид компиляции и интерпретации. Исходник сначала компилируется в байткод (платформо-независимое промежуточное представление), а потом виртуальная машина исполняет этот байткод и «на лету» компилирует «горячие» куски (часто исполняемые) в нативный машинный код, который кэшируется и используется в следующих вызовах.
Логика
  • Загрузка байткода в VM.
  • Сначала байткод интерпретируется — это даёт быстрый старт.
  • VM считает, сколько раз вызывается каждый кусок (метод, цикл).
  • Когда счётчик переваливает порог — кусок компилируется в нативный код и кэшируется.
  • Следующие вызовы того же куска идут уже через кэш — почти со скоростью полностью скомпилированного кода.
Преимущества
  • Гибкость интерпретации + скорость компиляции: горячие куски быстрые, холодные — гибкие.
  • Адаптивная оптимизация: JIT видит реальное поведение программы (какие типы реально приходят) и оптимизирует под них.
  • Портируемость на уровне байткода: один и тот же .class или .dll работает на любой ОС, где есть JVM/CLR.
Недостатки
  • «Прогрев» (warm-up): в начале запуск медленнее, пока JIT не успел скомпилировать горячие части.
  • JIT-компилятор сам потребляет CPU и память.
  • Машинный код, который JIT генерирует, всё равно привязан к текущей платформе — портируется именно байткод.
Примеры
  • Java + JVM (HotSpot, OpenJ9): .java → javac → .class (байткод) → JVM с JIT.
  • C# / .NET: .cs → csc → CIL/MSIL → CLR с JIT (RyuJIT).
  • JavaScript в браузерах: V8 (Chrome), SpiderMonkey (Firefox), JavaScriptCore (Safari) — все используют JIT.
  • PyPy — альтернативная реализация Python с JIT, в разы быстрее CPython на математике.

📦 D. Байткод-интерпретаторы (Bytecode interpreters)

Определение
Байткод (bytecode) — это набор инструкций низкого уровня, но платформо-независимый, спроектированный для эффективного исполнения программной виртуальной машиной. Это не машинный код реального CPU, а команды для «виртуального процессора» (например, JVM или Python VM).
Поток
  • Шаг 1. Исходный код компилируется в байткод (один раз, либо при первом запуске).
  • Шаг 2. Байткод-интерпретатор (виртуальная машина) читает эти байты и исполняет.
Где это видно
  • Python (CPython) — при импорте модуля или повторном запуске Python кэширует скомпилированный байткод в файлах .pyc (папка __pycache__). Это не ускоряет выполнение самого кода, но избавляет от повторной компиляции при следующем запуске.
  • Java — компилятор javac делает .class-файлы с байткодом, которые потом исполняет JVM.
  • Ruby (YARV) — Yet Another Ruby VM, исполняет байткод.
  • Erlang (BEAM) — виртуальная машина для байткода Erlang/Elixir.
Плюсы
  • Портируемость — байткод одинаков везде, где есть VM.
  • Быстрее «голой» интерпретации исходника — байткод уже разобран, не нужно повторно делать лексику/синтаксис.
  • Меньший размер — байткод компактнее исходного текста.
Минусы
  • Без JIT — всё равно медленнее, чем нативный машинный код.
  • Часть ошибок ловится только при исполнении конкретной инструкции байткода.
🧩 Где граница между байткодом и JIT? Сам по себе байткод-интерпретатор просто исполняет байты по очереди — без перевода в нативный код. JIT — это слой сверху, который замечает горячие участки и компилирует их в нативный код. Java и C# почти всегда идут с JIT; стандартный CPython — это «чистый» байткод-интерпретатор без JIT (а PyPy — с JIT).

6. Этапы компиляции (важно для экзамена)

Когда компилятор превращает исходник в машинный код (или интерпретатор — в байткод), он проходит через несколько чётко определённых этапов. На экзамене часто просят описать эту цепочку. Запомни её в таком порядке:

→

Токены → AST
→
3 Семантика Semantic analysis
Проверка смысла, типов
→
4 Оптимизация Optimization
Ускорение, упрощение
→
5 Генерация кода Code generation
AST → машинный код
Вход: исходный код score = 42 Выход: машинные инструкции 0110 0000 0100 0010

Этап 1 — Лексический анализ (Lexical analysis)

Компилятор разбивает исходный текст на токены (tokens): ключевые слова, идентификаторы, операторы, литералы, скобки. Это как «нарезать» предложение на отдельные слова и знаки препинания. На этом этапе ловятся ошибки вроде «неизвестный символ».

Пример. Для строки print("Hello World!") лексер выделит токены: print (идентификатор/ключевое слово), ( (открывающая скобка), "Hello World!" (строковый литерал), ) (закрывающая скобка).

Этап 2 — Синтаксический анализ / парсинг (Syntax analysis / parsing)

Поток токенов проверяется на соответствие грамматике языка. Парсер (parser) строит из токенов иерархическую структуру — дерево разбора (parse tree) или абстрактное синтаксическое дерево (AST, abstract syntax tree). Тут ловятся ошибки вроде «не хватает закрывающей скобки» или «после if нет условия».

Например, выражение print("Hello World!") должно соответствовать структуре «функциональный вызов»: имя функции, за ним скобки, в скобках — аргументы. Если эта структура нарушена — синтаксическая ошибка.

Этап 3 — Семантический анализ (Semantic analysis)

Парсер уже знает структуру кода. Семантика — это его смысл. Здесь компилятор проверяет:

  • Существует ли вызываемая функция (print — существует).
  • Совпадают ли типы (нельзя сложить число и список без явного преобразования).
  • Объявлены ли все переменные до использования.
  • Совместимы ли типы аргументов с сигнатурой функции.

Это самый «умный» из ранних этапов. Семантические ошибки звучат как «variable not declared» или «type mismatch: expected int, got string».

Этап 4 — Оптимизация (Optimization)

Компилятор улучшает код, не меняя его поведение: переставляет инструкции, удаляет недостижимый код, разворачивает циклы, сворачивает константы (2 + 3 сразу → 5), инлайнит маленькие функции. Цель — чтобы исполнение шло быстрее или потребляло меньше памяти. У интерпретаторов этот этап обычно ограничен или отсутствует.

Этого этапа нет в интерпретаторах в полном объёме — или он минимальный. Полноценная оптимизация — преимущество именно компиляторов (и JIT, который оптимизирует адаптивно).

Этап 5 — Генерация кода (Code generation)

Финальная стадия. Оптимизированное промежуточное представление превращается в:

  • Машинный код для конкретной архитектуры — у «полных» компиляторов (C, Rust, Go).
  • Байткод — у интерпретаторов и гибридных систем (Python .pyc, Java .class).

После этого код готов к исполнению: либо ОС запускает бинарь напрямую, либо виртуальная машина / интерпретатор начинает исполнять байткод.

🎯 Запомни мнемонику: «Лена Сидит Семантически и Оптимально Готовит код» — Лексика → Синтаксис → Семантика → Оптимизация → Генерация кода. В интерпретаторе те же первые три этапа, затем сразу генерация байткода и исполнение — без полноценной оптимизации.

7. Линкеры и загрузчики (Linkers & Loaders)

После того, как компилятор превратил каждый .c или .cpp файл в объектный код (.o, .obj) — это ещё не готовая программа. Современные программы состоят из множества файлов и используют стандартные библиотеки (standard library) и сторонние библиотеки.

🔗 Линкер (Linker)

Что делает
Берёт все объектные файлы и нужные библиотеки, склеивает их в один исполняемый файл. Разрешает ссылки между файлами: если в одном файле вызывается printf, линкер находит её в стандартной библиотеке C и подставляет правильный адрес.
Виды линковки
  • Статическая: библиотека «вшивается» внутрь .exe → файл больше, но не нужны внешние зависимости.
  • Динамическая: .exe ссылается на .dll (Windows) или .so (Linux) → файл меньше, но при отсутствии библиотеки программа не запустится.

📥 Загрузчик (Loader)

Что делает
Часть ОС: когда пользователь запускает программу, загрузчик читает исполняемый файл с диска, размещает его в памяти, настраивает регистры, передаёт управление на первую инструкцию (entry point).
Этапы
  • Читает заголовок .exe / ELF.
  • Выделяет адресное пространство процесса.
  • Копирует код и данные в нужные области памяти.
  • При динамической линковке загружает .dll / .so в память.
  • Передаёт управление CPU на стартовый адрес.

8. Когда какой подход выбирать

Программа существует ради задачи. Выбор подхода зависит от того, что важнее: скорость выполнения, скорость разработки, портируемость или баланс.

Интерпретатор

Быстрая разработка и тестирование

Стартап-команда прототипирует IoT-приложение для управления умными устройствами в реальном времени. Python или JavaScript позволяют быстро менять код и сразу видеть результат, без шага компиляции. На раннем этапе важна гибкость, а не скорость выполнения.

Компилятор

Производительность критична

Платформа высокочастотного трейдинга (HFT), где сделки исполняются за микросекунды. C++, Rust или Go компилируются в нативный машинный код, который CPU исполняет напрямую. Игры и VR-приложения — та же логика: каждый кадр должен укладываться в 16 миллисекунд.

JIT + байткод

Кроссплатформенный enterprise-софт

Компания пишет CRM-систему, которая должна работать на Windows, macOS и Linux. Java компилируется в байткод, который JVM исполняет на любой ОС. Производительность достаточно высокая благодаря JIT-компиляции в JVM. То же относится к C# и .NET.

Интерпретатор

Скрипты и автоматизация

Системному администратору нужно автоматизировать ежедневные задачи: парсинг логов, копирование файлов, отправка отчётов. Python или Bash — идеальный выбор: маленькие скрипты, без сложной сборки, легко править.

Компилятор

Системное программирование

Операционные системы, драйверы устройств, встраиваемые системы. Нужен прямой доступ к памяти, предсказуемое поведение, минимальные накладные расходы. C и Rust — стандарт индустрии (ядро Linux на C, всё больше частей на Rust).

JIT

Веб-фронтенд

JavaScript в браузере. Когда-то был чисто интерпретируемым, сейчас все движки (V8, SpiderMonkey, JavaScriptCore) используют JIT — это позволило построить сложные веб-приложения вроде Google Docs или Figma со скоростью, близкой к нативным.

9. Связанные понятия

Стандартная библиотека (Standard library)

Готовый набор модулей, классов и функций, который поставляется вместе с языком. У Python это os, json, math и сотни других. У Java это java.util, java.io. Стандартная библиотека сильно облегчает разработку: не нужно реализовывать всё с нуля.

Парсер (Parser)

Программный компонент, который анализирует поток входных токенов по правилам грамматики и строит из них иерархическое представление — дерево разбора или AST. Парсеры используются не только в компиляторах: HTML-парсер в браузере, JSON-парсер в API, SQL-парсер в БД — все они работают по той же логике.

Семантика (Semantics)

Точный смысл, связанный с конкретной программной конструкцией: «что делает эта инструкция?». В отличие от синтаксиса (как должно быть написано), семантика отвечает за что произойдёт при исполнении. Например, синтаксически правильный код x = y + z семантически ошибочен, если y — это строка, а z — число (в строгом языке).

Бинарный исполняемый файл (Binary executable)

Файл, который содержит готовые машинные инструкции в формате, понятном конкретной ОС (PE для Windows, ELF для Linux, Mach-O для macOS). ОС умеет загружать такой файл в память и сразу передавать управление CPU.

10. Тренировочные вопросы (нажми, чтобы увидеть подсказку к ответу)

Вопросы — из стиля экзамена IB (включая Practice questions 49–51 из учебника и расширенные авторские). Ответы — подсказки в моём изложении, а не официальные mark schemes.

Вопрос 1. Опиши, как высокоуровневая инструкция переводится в машинный код. 4 балла

Что писать (4 ключевых пункта):

  • Лексический анализ: исходный текст разбивается на токены (ключевые слова, операторы, идентификаторы, литералы).
  • Синтаксический анализ (парсинг): токены собираются в дерево разбора (AST) согласно грамматике языка; проверяется правильность структуры.
  • Семантический анализ: проверяется смысл — типы, объявленные переменные, существующие функции.
  • Генерация кода: AST превращается в машинный код для конкретной архитектуры процессора (после промежуточной оптимизации в случае компилятора).

Каждый этап = 1 балл. Если упомянуть оптимизацию пятым шагом — это плюс, но не обязательно для 4 баллов.

Вопрос 2. Обозначь роль парсера в процессе трансляции языков программирования. 2 балла

Что писать:

  • Парсер анализирует поток токенов (полученный из лексического анализатора) согласно грамматическим правилам языка.
  • Он строит иерархическую структуру — дерево разбора или абстрактное синтаксическое дерево (AST), показывающее, как элементы программы связаны между собой.

2 пункта = 2 балла. Можно добавить, что парсер ловит синтаксические ошибки (несовпадение скобок и т.д.), но это не обязательно.

Вопрос 3. Оцени использование байткода в трансляции языков программирования. 3 балла

Что писать (с балансом плюсов и минусов — «evaluate»):

  • Плюс — портируемость: один и тот же байткод исполняется на любой платформе, где есть соответствующая виртуальная машина (один .class на Windows, macOS, Linux).
  • Плюс — компромисс скорости: байткод-интерпретация быстрее, чем интерпретация исходника напрямую, потому что лексика и синтаксис уже разобраны (а с JIT — почти на уровне нативного кода).
  • Минус — нужна виртуальная машина: без VM байткод не исполняется; есть накладные расходы по памяти и старту, и всё равно медленнее, чем чисто скомпилированный нативный код без JIT.

«Evaluate» предполагает баланс — обязательно укажи и сильные, и слабые стороны.

Вопрос 4. Сравни и противопоставь компилятор и интерпретатор по четырём критериям: время трансляции, обнаружение ошибок, портируемость, применимость. 8 баллов

Что писать (по 2 балла за каждый критерий — указать оба подхода):

  • Время трансляции.
    Компилятор: переводит весь исходник один раз до запуска; во время исполнения трансляции нет, но цикл «правка → компиляция → запуск» долгий.
    Интерпретатор: нет отдельной стадии компиляции; перевод происходит построчно во время выполнения — старт мгновенный, но каждое исполнение тратит время на трансляцию.
  • Обнаружение ошибок.
    Компилятор: большинство ошибок (синтаксис, типы, неопределённые переменные) ловятся до запуска — программа просто не соберётся.
    Интерпретатор: ошибки всплывают во время исполнения, в момент, когда строка реально достигнута. Опечатка в редко вызываемой ветке может не проявиться при тестировании.
  • Портируемость.
    Компилятор: исполняемый файл привязан к ОС и архитектуре CPU; для каждой платформы нужна отдельная сборка.
    Интерпретатор: один и тот же исходник работает на любой ОС, где установлен интерпретатор — высокая портируемость.
  • Применимость.
    Компилятор: подходит для производительно-критичных приложений — игры, ОС, системное ПО, HFT.
    Интерпретатор: подходит для скриптинга, веб-разработки, прототипирования, обучения, где важна скорость итераций.

8 баллов = 4 критерия × 2 балла (по 1 баллу за каждый подход в критерии).

Вопрос 5. Объясни, как JIT-компиляция сочетает преимущества компиляторов и интерпретаторов. 6 баллов

Что писать:

  • JIT принимает на вход не исходник, а байткод — промежуточное платформо-независимое представление, скомпилированное заранее (например, .class у Java).
  • Сначала виртуальная машина интерпретирует байткод — это даёт быстрый старт, как у обычного интерпретатора.
  • VM отслеживает «горячие» участки кода (методы и циклы, которые вызываются много раз).
  • Когда счётчик превышает порог, JIT компилирует эти участки в нативный машинный код и кладёт в кэш.
  • Следующие вызовы того же кода идут уже через кэш — почти со скоростью полностью скомпилированной программы.
  • Бонус: JIT может оптимизировать адаптивно — видит реальные типы и шаблоны выполнения, чего статический компилятор сделать не может.

6 баллов: каждый чёткий пункт ≈ 1 балл. Обязательно упомянуть «горячие участки» и кэш — это ключевая идея JIT.

Вопрос 6. Различи скомпилированные программы на C++ и интерпретируемые на Python с точки зрения портируемости. 4 балла

Что писать (по контрасту):

  • C++: компилятор создаёт бинарный файл, привязанный к конкретной ОС и архитектуре CPU. Бинарь под Windows x86-64 не запустится на Linux ARM — нужна отдельная компиляция под каждую целевую платформу.
  • Python: исходный код — обычный текст. Он работает на любой ОС, где установлен Python-интерпретатор. Разработчику не нужно собирать отдельные версии для Windows, macOS и Linux.
  • Следствие: C++ — низкая портируемость, высокая скорость. Python — высокая портируемость, более низкая скорость.
  • Дополнительно: чтобы запустить C++ программу, конечному пользователю достаточно бинаря; чтобы запустить Python — нужен установленный интерпретатор (или упаковка через PyInstaller).
Вопрос 7. Назови и кратко опиши пять этапов компиляции. 5 баллов

Что писать (по 1 баллу за этап с коротким описанием):

  • Лексический анализ: исходный код разбивается на токены (ключевые слова, операторы, литералы).
  • Синтаксический анализ: токены проверяются на соответствие грамматике; строится AST (абстрактное синтаксическое дерево).
  • Семантический анализ: проверяется смысл — типы, объявления переменных, существование функций.
  • Оптимизация: код упрощается и ускоряется без изменения поведения (свёртка констант, удаление мёртвого кода).
  • Генерация кода: AST превращается в машинный код (или байткод) для целевой платформы.
Вопрос 8. Объясни, почему интерпретатор может «не заметить» ошибку в редко вызываемой части программы, а компилятор — заметит. 4 балла

Что писать:

  • Компилятор обрабатывает весь исходник целиком до запуска. Он проходит лексику, синтаксис и семантику по каждой строке — в том числе по тем, что лежат в редко используемых ветках. Поэтому опечатка типа def calcualte(): вместо calculate будет найдена сразу при сборке.
  • Интерпретатор обрабатывает только те строки, которые реально исполняются. Если ветка if rare_condition: не выполнилась во время теста — её содержимое никогда не было проанализировано.
  • Следствие: в Python ошибки могут «дожить» до продакшена, если код не покрыт тестами. В C++ или Rust — нет, программа просто не соберётся.
  • Поэтому в интерпретируемых языках критично иметь полное покрытие тестами и/или использовать линтеры (например, mypy, pylint, flake8), которые статически анализируют код без запуска.
Вопрос 9. Какой подход к трансляции ты бы выбрал для (a) операционной системы и (b) образовательного скрипта для школьников? Обоснуй. 6 баллов

Что писать:

  • (a) Операционная система — компилятор (C или Rust).
    • ОС работает в самом низком слое и должна быть максимально быстрой — каждый системный вызов критичен.
    • Нужен прямой доступ к памяти и регистрам CPU — это есть только в скомпилированных языках низкого уровня.
    • На стадии загрузки ОС нет «другого интерпретатора», который мог бы её исполнить — нужен нативный машинный код.
  • (b) Образовательный скрипт — интерпретатор (Python).
    • Школьник может сразу видеть результат — нет шага компиляции, что снижает порог входа.
    • Синтаксис Python близок к псевдокоду, что облегчает обучение.
    • Один и тот же файл работает на Windows, macOS, Linux — учителю не нужно собирать разные версии для разных школьных компьютеров.
    • Скорость выполнения не важна — учебный скрипт обычно небольшой.

3 балла за (a), 3 балла за (b). Главное — обосновать каждый выбор минимум двумя аргументами.

Вопрос 10. Объясни роль линкера и загрузчика в превращении исходного кода в работающую программу. 4 балла

Что писать:

  • Линкер (linker) берёт объектные файлы (результат компиляции каждого исходника отдельно) и склеивает их в единый исполняемый файл; разрешает ссылки между файлами (если в одном вызывается функция из другого).
  • Линкер также подключает библиотеки — статически (вшивает код внутрь .exe) или динамически (оставляет ссылку на .dll/.so).
  • Загрузчик (loader) — часть ОС: при запуске программы читает .exe с диска, размещает в памяти, настраивает регистры, передаёт управление на entry point.
  • Без линкера у тебя только разрозненные .o-файлы; без загрузчика готовый .exe не будет исполняться — это два разных, но необходимых шага между «скомпилировано» и «работает».

11. Практическое задание

🔬 Сравни три подхода на одной задаче

Цель: прочувствовать разницу между компилятором, интерпретатором и JIT на одной и той же программе.

Сценарий: возьми классическую программу «вычисли N-е число Фибоначчи рекурсивно» (намеренно медленный алгоритм). Запусти три версии: на C++, Python и Java. Замерь время для N = 35.

Инструкции:

  1. Напиши одну и ту же рекурсивную функцию Фибоначчи на трёх языках: C++, Python, Java.
  2. C++. Скомпилируй командой g++ -O2 fib.cpp -o fib, запусти time ./fib. Запиши время.
  3. Python. Просто запусти time python fib.py. Запиши время.
  4. Java. Скомпилируй javac Fib.java (получишь Fib.class — байткод). Запусти time java Fib. Запиши время.
  5. Сравни три числа. Обычно: C++ — десятые доли секунды, Java — секунды (но быстрее Python после прогрева JIT), Python — десятки секунд.
  6. Объясни в одном абзаце, почему такая разница. Используй термины: нативный машинный код, JIT, интерпретация байткода, динамическая типизация.

Что важно запомнить

  • Трансляция = превращение человеко-понятного кода в команды для CPU. Три основных подхода: компилятор, интерпретатор, JIT.
  • Компилятор переводит весь код один раз → быстрое исполнение, ранние ошибки, низкая портируемость. Языки: C, C++, Rust, Go.
  • Интерпретатор переводит строка за строкой во время выполнения → медленное исполнение, поздние ошибки, высокая портируемость. Языки: Python, JavaScript, Ruby, PHP.
  • JIT = байткод + компиляция «на лету» горячих участков → гибрид. Языки: Java, C#, JavaScript (V8), PyPy.
  • Байткод — платформо-независимое промежуточное представление, исполняется виртуальной машиной (JVM, CLR, Python VM, BEAM).
  • Пять этапов компиляции: Лексика → Синтаксис → Семантика → Оптимизация → Генерация кода.
  • Парсер строит AST из токенов и проверяет соответствие грамматике; семантика отвечает за смысл (типы, существование функций).
  • Линкер склеивает объектные файлы и библиотеки в один .exe; загрузчик (часть ОС) запускает .exe в памяти.
  • На экзамене вопрос «evaluate» — это всегда сильные и слабые стороны. «Distinguish» — контраст по чётким критериям (время, ошибки, портируемость, применимость).
  • Машинный код — это бинарные команды, привязанные к архитектуре CPU (x86, ARM, MIPS). Ассемблер — почти один-к-один с машинным кодом, но с читаемыми мнемониками (MOV, ADD, JMP).
