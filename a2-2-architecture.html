Архитектура сетей

Эта тема (A2.2) — про то, как сеть устроена как целое: какой формы она «нарисована» на схеме (топология), какие специализированные компьютеры в ней работают (серверы), кто кому что подаёт (клиент-сервер vs P2P) и как её разделить на части ради безопасности и скорости (сегментация). Понимание архитектуры — это понимание почему сеть устойчива (или хрупка), быстра (или тормозит), безопасна (или дырявая).

Хочешь сразу пощупать топологии? Открой интерактивный симулятор топологий — там можно построить звезду / шину / кольцо / mesh / гибрид и «убить» узел, чтобы посмотреть, как сеть реагирует на отказ.

🕸️ A2.2.1 — Топологии сетей

Топология сети — это расположение устройств (узлов, nodes) и связей (links) внутри сети. Топология может быть:

  • Физическая — фактическая раскладка кабелей, проводов, маршрутизаторов, коммутаторов и других «железок».
  • Логическая — как данные текут по сети, независимо от физического расположения. Например: физически у тебя звезда, но логически (по поведению трафика) может быть кольцо.

Для экзамена важно знать характеристики каждой топологии по 5 осям: надёжность, скорость, масштабируемость, поведение при отказе, стоимость. И — где её реально применяют.

⭐ Звезда (Star topology)

Каждое устройство соединено напрямую с центральным устройством (обычно с коммутатором или маршрутизатором). Получается «звёздочка» с центром.

Плюсы
  • Надёжность каждого узла — обрыв одного кабеля или поломка одного устройства не уронит сеть, пострадает только этот узел.
  • Скорость — современные коммутаторы минимизируют задержки; прямая связь с центром убирает «бутылочные горлышки».
  • Лёгкая масштабируемость — добавить новое устройство = подключить кабель в свободный порт.
  • Коллизии данных почти исключены (коммутатор управляет трафиком).
Минусы
  • Единая точка отказа — если ляжет центральный коммутатор/хаб, ляжет вся сеть.
  • У коммутатора ограниченное число портов — это потолок масштабирования.
  • Дороже по кабелям — у каждого устройства своя выделенная линия в центр.

Где применяют: 🏠 Домашние сети (роутер + Wi-Fi точка в центре), 🏢 офисные LAN с центральным управляемым коммутатором.

🚌 Шина (Bus topology)

Все устройства подключены к одному общему кабелю (магистрали, backbone). Каждое устройство «слушает» весь трафик и берёт только адресованные ему пакеты. На концах кабеля стоят терминаторы, чтобы сигнал не отражался.

Плюсы
  • Дёшево — один кабель на всех, мало проводки.
  • Просто проектировать для маленьких сетей.
Минусы
  • Один обрыв магистрали = вся сеть мертва.
  • Коллизии — два устройства, передающие одновременно, сталкиваются, и приходится повторять. При большом числе узлов производительность падает.
  • Плохая безопасность — все «слышат» всё.
  • Сложно искать поломку (где обрыв на длинном кабеле?).

Где применяют: почти не используется в современных сетях. Встречается в легаси Ethernet-сегментах (10BASE-2/10BASE-5) и в некоторых промышленных шинах (CAN bus в автомобилях — концептуально похоже).

⭕ Кольцо (Ring topology)

Каждое устройство соединено с двумя соседями — слева и справа. Данные ходят по кольцу в одном (или двух) направлении(ях), пока не дойдут до адресата. Часто используется механизм токена: у кого токен — тот сейчас передаёт.

Плюсы
  • Нет коллизий при использовании токена — только один передаёт за раз.
  • Производительность предсказуемая и стабильная под нагрузкой.
  • Дёшево по кабелю (меньше проводки, чем у звезды).
Минусы
  • Обрыв одного звена = всё кольцо ломается (если кольцо одностороннее).
  • Если выходит из строя один узел — данные дальше не идут.
  • Добавить устройство = временно «разорвать кольцо».
  • Большая задержка: пакет может пройти много узлов до адресата.

Где применяют: исторически Token Ring (IBM, 1980-е), FDDI (оптические бэкбоны университетов). Сейчас редко, но идея «двойного кольца» используется в SONET/SDH и метро-Ethernet для резервирования.

🕸️ Mesh (Сетчатая топология)

Каждый узел (node) соединён с как можно большим числом других узлов. Полный mesh = каждый соединён с каждым. Частичный mesh = только с некоторыми. Получается «паутина» из множества прямых связей.

Плюсы
  • Самая высокая надёжность. Если узел или линия упали, данные автоматически идут другим путём (избыточность маршрутов).
  • Нет единой точки отказа.
  • Лёгкая масштабируемость: добавил узел, включил питание — mesh-протоколы сами найдут соседей.
  • Умные алгоритмы маршрутизации находят оптимальный путь — мало коллизий.
Минусы
  • Очень дорого по кабелям и портам (если проводной). В полном mesh связей растёт как n(n-1)/2.
  • Скорость может страдать на «длинных» путях из-за множества хопов (hops — переходов между узлами).
  • Корпоративные mesh-узлы дороги.

Где применяют: 📶 современные домашние Wi-Fi mesh-системы (Google Nest Wifi, Eero, Amazon eero, TP-Link Deco). Городские mesh-сети для интернета в зонах со слабой инфраструктурой. IoT-сенсоры на складах и заводах. Военные тактические сети (нет «единого центра» — труднее уничтожить).

🌳 Дерево (Tree topology)

Иерархическая топология: один корневой узел (root) разветвляется на под-узлы, те — на свои под-узлы. Сочетает шину (магистраль вверху) и звёзды (ветки внизу). По сути — расширенная звезда с уровнями.

Плюсы
  • Иерархия = простой менеджмент. Удобно для крупных кампусов, корпоративных сетей с этажами/зданиями.
  • Хорошая масштабируемость — можно расти, добавляя ветки.
  • Сегментация естественная — каждая ветка ≈ отдел.
Минусы
  • Если упал корневой узел — лежит вся сеть.
  • Отказ промежуточного коммутатора отрезает всё «ниже» по ветке.
  • Сложно настроить и поддерживать в больших масштабах.
  • Дорого по кабелям и оборудованию.

Где применяют: 🏫 университетские кампусы, корпоративные сети «штаб-квартира → офисы → этажи → компьютеры», провайдеры доступа в интернет (магистраль → региональные узлы → клиенты).

🧬 Гибрид (Hybrid topology)

Сочетание двух и более топологий ради того, чтобы взять у каждой её лучшие свойства и обойти слабые места. Например: звезда для ядра (главные серверы и роутеры) + mesh для периферии (для устойчивости).

Плюсы
  • Очень высокая надёжность — комбинирует резервные пути mesh с управляемостью звезды.
  • Гибкая масштабируемость — расширять можно через те части, которые легче масштабировать (mesh или звезда).
  • Можно подобрать оптимальные характеристики по производительности для разных частей сети.
Минусы
  • Сложно проектировать и обслуживать — несколько технологий = больше нюансов.
  • Дороже простых решений из-за управления и поддержки разных топологий.
  • Требует «умных» алгоритмов маршрутизации, чтобы оптимизировать пути.

Где применяют: 🏢 крупные корпоративные сети («звезда» в офисе + mesh для удалённых станций), 🏙️ smart cities (интеграция разных служб: транспорт, освещение, видеонаблюдение).

💡 Экзаменационный лайфхак: если просят «сравни две топологии», структурируй ответ по 5 осям — надёжность, скорость, масштабируемость, поведение при отказе, стоимость. Это даёт чёткие баллы за каждый пункт.

Тренировочные вопросы — A2.2.1 Топологии

Вопрос 1. Назови одно преимущество использования звёздной топологии при проектировании сети. 1 балл

Что писать (любой 1 пункт):

  • Высокая надёжность узлов: отказ одного устройства или кабеля не уронит сеть — пострадает только этот узел.
  • Лёгкая масштабируемость: новое устройство добавляется простым подключением к свободному порту центрального коммутатора.
  • Хорошая производительность: прямая связь устройства с центром минимизирует задержки и коллизии.
Вопрос 2. Опиши, в чём основное различие между физической и логической сетевой топологией. 2 балла

Что писать:

  • Физическая топология — это фактическое расположение кабелей, проводов, коммутаторов и других «железок» — то, что видно глазами.
  • Логическая топология — это как данные текут по сети, независимо от физического расположения устройств.
  • Можно иметь физическую звезду (все подключены к одному коммутатору), но логически — кольцо или шину, в зависимости от протокола.

2 балла = по 1 за каждое определение/контраст.

Вопрос 3. Опиши два признака mesh-топологии. 4 балла

Что писать (2 признака × 2 балла каждое):

  • Высокая надёжность за счёт избыточности. Если один узел или соединение выходит из строя, данные автоматически перенаправляются другими путями. Нет единой точки отказа.
  • Прямые соединения многих узлов. В полном mesh каждый узел соединён с каждым; в частичном — с некоторыми. Это даёт умной маршрутизации возможность найти кратчайший/менее загруженный путь.
  • Лёгкая масштабируемость: большинство mesh-протоколов умеют автонастройку — добавил устройство, включил, и оно само нашло соседей.
  • Скорость зависит от числа «хопов» между узлами и от нагрузки на сеть.
Вопрос 4. (a) Опиши преимущества гибридной топологии с примерами из учебника. (b) Опиши реальное применение гибридной топологии. 4 + 2 балла

(a) Преимущества (4 балла):

  • Высокая надёжность — благодаря множеству путей передачи данных сеть остаётся работоспособной даже при отказе компонентов. Это сочетание избыточности mesh с централизованным управлением звезды.
  • Гибкая производительность — прямые связи в одних сегментах (как в mesh) увеличивают скорость и уменьшают число хопов; централизованные элементы упрощают маршрутизацию и управление.
  • Гибкая масштабируемость — добавлять новые узлы можно через те части, которые проще расширять (mesh- или звёздные сегменты).
  • Использование умных алгоритмов маршрутизации — позволяет оптимизировать пути данных через разные топологии, уменьшая коллизии и заторы.

(b) Применение (2 балла):

  • Крупные корпоративные сети: звезда для центральных офисов + mesh для соединения удалённых рабочих станций и устройств.
  • Smart cities: гибрид интегрирует разные сервисы и инфраструктуру (транспорт, освещение, безопасность), обеспечивая устойчивость, гибкость и покрытие.
Вопрос 5. Объясни поведение шинной и кольцевой топологии при обрыве кабеля. 4 балла

Что писать (по 2 балла на каждую):

  • Шина: весь трафик идёт по одному общему кабелю. Обрыв магистрали = полная потеря связи для всех устройств. Без терминаторов на концах сигнал ещё и отражается, вызывая помехи.
  • Кольцо: данные обходят круг через всех соседей. Обрыв одного звена = поток ломается, передача останавливается (для одностороннего кольца). В двойном кольце (FDDI) сеть может «свернуться» вокруг обрыва.

🖥️ A2.2.2 — Функции серверов HL

Сервер — это компьютер (аппаратура или софт), который предоставляет данные, ресурсы или услуги другим компьютерам (клиентам) по сети. Технически сервером может быть любой компьютер, но в реальности серверы обычно отличаются от обычных ПК: они построены для высокой надёжности, отказоустойчивости и стабильной производительности. Серверы работают 24/7, часто стоят в стойках в дата-центрах, у них продублированы блоки питания, диски (RAID) и сетевые карты.

🗂️ A. DNS-сервер (Domain Name System)

Назначение
Переводит понятные человеку доменные имена (например, www.google.com) в числовые IP-адреса (например, 172.217.16.14). Без этого перевода маршрутизация пакетов невозможна.
Как работает
  • Браузер запрашивает IP у локального DNS (обычно у провайдера).
  • Если локальный DNS не знает ответа — он рекурсивно опрашивает корневые серверы → серверы TLD (например, для .com) → авторитетный DNS домена, у которого есть точный ответ.
  • Найденный IP кэшируется на время TTL (Time To Live), чтобы ускорить будущие запросы.
Надёжность
DNS-серверы географически распределены и реплицируются. Если один сервер упал — другие продолжают отвечать. Крупные провайдеры используют балансировку нагрузки между серверами.

Пример: ты вводишь google.com → DNS возвращает 142.250.190.46 → твой браузер обращается уже к этому IP.

📡 B. DHCP-сервер (Dynamic Host Configuration Protocol)

Назначение
Автоматически выдаёт устройствам в сети IP-адрес и другие сетевые настройки (маску подсети, шлюз по умолчанию, DNS-серверы), чтобы они могли начать общаться с другими сетями.
Процесс DORA
  • Discovery — клиент рассылает DHCPDISCOVER, чтобы найти DHCP-сервер.
  • Offer — сервер отвечает DHCPOFFER с предложенным IP.
  • Request — клиент подтверждает: «беру этот IP» (DHCPREQUEST).
  • Acknowledgment — сервер закрепляет аренду (DHCPACK) на определённое время (lease).
Режимы
  • Динамический: IP даётся из пула на ограниченное время. По истечении аренды клиент запрашивает заново.
  • Статический: конкретный IP закрепляется за устройством по его MAC. Нужно для принтеров, серверов, оборудования, которому нужен постоянный адрес.

Зачем нужен: без DHCP админ должен вручную назначать IP каждому новому устройству — это долго и легко допустить ошибку. Большие сети часто имеют несколько DHCP-серверов для отказоустойчивости и балансировки.

📁 C. Файловый сервер (File server)

Назначение
Централизованно хранит и управляет файлами в сети, позволяя нескольким пользователям и устройствам обращаться к ним и делиться ими.
Что делает
  • Клиент отправляет запрос на файл → сервер проверяет права доступа → разрешает или запрещает.
  • При разрешении — пользователь может читать, редактировать или сохранять файл прямо на сервере (в зависимости от прав).
  • Использует протоколы SMB (Windows) или NFS (Unix/Linux).
Надёжность
Для отказоустойчивости используют RAID (избыточные массивы дисков) и регулярные резервные копии (backup). Это защищает данные от потери при отказе диска или повреждении файлов.

Пример: в компании файловый сервер хранит общие документы — сотрудники открывают их с любых устройств в офисе или удалённо. Нет нужды в флешках и пересылке файлов почтой.

✉️ D. Почтовый сервер (Mail server)

Назначение
Отправляет, принимает и хранит электронную почту. Это как «цифровое почтовое отделение». Может быть в роли MTA (Mail Transfer Agent, перевозчик писем) или MDA (Mail Delivery Agent, доставщик в ящик).
Протоколы
  • SMTP (Simple Mail Transfer Protocol) — для отправки писем.
  • POP3 — клиент скачивает письма на устройство и обычно удаляет с сервера.
  • IMAP — клиент синхронизирует письма; они остаются на сервере, доступны с любого устройства.
Безопасность
  • Спам-фильтры и антивирус — защита от вредоносных писем.
  • DKIM, SPF, DMARC — современные политики аутентификации отправителя, без которых письмо чаще попадёт в спам.

Пример: когда ты отправляешь письмо с gmail.com → твой клиент подключается к SMTP-серверу Google → тот определяет почтовый сервер получателя по доменной части адреса и пересылает туда → у получателя письмо забирается через IMAP/POP3.

🚦 E. Прокси-сервер (Proxy server)

Назначение
Работает как посредник между устройством пользователя и интернетом. Ты не идёшь к сайту напрямую — твой запрос идёт через прокси, который сам сходит и принесёт ответ.
Ключевые роли
  • Конфиденциальность и безопасность: скрывает реальный IP пользователя, предотвращает прямые атаки на внутреннюю сеть.
  • Фильтрация контента: в школе/офисе блокирует сайты, не соответствующие политике (соцсети, NSFW и т.д.).
  • Кэширование: хранит копии часто запрашиваемых ресурсов — ускоряет доступ и экономит трафик.
  • Логирование: ведёт журнал всех веб-запросов; позволяет отследить вредоносную или подозрительную активность.

Пример: в офисе сотрудник заходит на сайт → запрос идёт на прокси → прокси проверяет политику → если сайт разрешён, прокси берёт страницу (или отдаёт из кэша) и возвращает сотруднику. Если запрещён — выдаёт сообщение об отказе.

Публичные прокси позволяют пользователям скрыть свой IP и геолокацию — это полезно для доступа к гео-ограниченному контенту и обхода цензуры.

🌐 F. Веб-сервер (Web server)

Назначение
Хранит, обрабатывает и отдаёт веб-страницы (HTML, CSS, JavaScript, картинки) клиентам по запросу через протоколы HTTP/HTTPS.
Как работает
  • Хранит файлы сайта (HTML, CSS, JS, медиа).
  • Получает HTTP-запрос от браузера, ищет нужный файл, отдаёт его. Если страница динамическая — сначала выполняет серверный скрипт (PHP, Python, Node.js) и генерирует HTML «на лету».
  • Часто использует SSL/TLS для шифрования (HTTPS).
Популярные
Apache, Nginx, IIS (Microsoft).

Пример: «Bekka's Bakery» хостит сайт на веб-сервере. Клиент вводит URL → браузер посылает HTTP-запрос → сервер находит файл меню (или запускает скрипт и тянет меню из БД) → возвращает HTML → браузер рендерит страницу.

🗄️ G. Сервер баз данных (Database server)

Назначение
Хранит структурированные данные (таблицы, записи) и обрабатывает запросы клиентов на чтение, запись, обновление и удаление данных. Использует SQL или специализированные API.
Что делает
  • Принимает SQL-запросы (например, SELECT * FROM users WHERE id=42) и возвращает результат.
  • Контролирует транзакции (ACID-свойства): данные либо записываются полностью, либо не записываются вовсе.
  • Управляет правами доступа: кто может читать/писать какие таблицы.
  • Делает резервные копии и реплицирует данные на другие серверы для отказоустойчивости.

Пример: сервер MySQL/PostgreSQL хранит данные пользователей соцсети. Веб-сервер (или application-сервер) отправляет ему SQL-запрос, получает данные и формирует страницу.

⚙️ H. Сервер приложений (Application server)

Назначение
Выполняет бизнес-логику приложения — то есть «думающий» слой между клиентом (браузером/мобилкой) и хранилищем данных (БД).
Что делает
  • Обрабатывает действия пользователя: «добавь в корзину», «рассчитай стоимость доставки», «прими платёж».
  • Связывает запросы клиента с базой данных и с другими сервисами.
  • Применяет правила бизнеса (скидки, проверки, авторизация).
  • Часто работает в трёхслойной архитектуре: клиент ↔ application server ↔ database server.

Пример: когда ты оформляешь заказ в интернет-магазине, веб-сервер отдаёт страницу, application-сервер обрабатывает «положить в корзину» и «оплатить», а database-сервер хранит товары и заказы.

Тренировочные вопросы — A2.2.2 Серверы (HL)

Вопрос 6. Назови основную функцию DNS-сервера. 1 балл

Что писать:

  • DNS-сервер переводит человекочитаемые доменные имена (например, www.google.com) в соответствующие числовые IP-адреса (например, 172.217.16.14), что позволяет маршрутизации пакетов работать.
Вопрос 7. Опиши роль DHCP-сервера. 2 балла

Что писать (по 1 баллу на пункт):

  • DHCP-сервер автоматически назначает IP-адреса и другие сетевые параметры (маску подсети, шлюз по умолчанию, адрес DNS-сервера) устройствам, подключающимся к сети.
  • Это упрощает администрирование — не нужно настраивать каждое устройство вручную; снижает риск конфликтов IP-адресов.
Вопрос 8. Опиши две функции почтового сервера. 4 балла

Что писать (любые 2 × 2 балла):

  • Отправка писем по протоколу SMTP: клиент подключается к SMTP-серверу, тот определяет почтовый сервер получателя по доменной части адреса и пересылает письмо туда.
  • Получение и хранение писем по протоколу POP3 (скачивание и обычно удаление с сервера) или IMAP (синхронизация, письма остаются на сервере и доступны с разных устройств).
  • Защита от спама и вредоносного ПО: спам-фильтры, антивирус, политики DKIM/SPF/DMARC для аутентификации отправителя.
  • Маршрутизация писем внутри организации: если получатель — на том же сервере, письмо доставляется прямо в его ящик без выхода в интернет.
Вопрос 9. Опиши преимущества и потенциальные недостатки использования прокси-сервера в организации. 4 балла

Преимущества (по 1 баллу):

  • Контроль и фильтрация контента: можно блокировать нежелательные сайты согласно политике организации.
  • Безопасность: прокси маскирует реальные IP-адреса внутренних устройств и предотвращает прямые атаки.
  • Кэширование: часто запрашиваемые ресурсы хранятся локально — экономия трафика и ускорение доступа.
  • Мониторинг и логи: можно отследить вредоносную или нецелевую активность.

Недостатки (по 1 баллу):

  • Единая точка отказа: если прокси упадёт, никто не сможет выйти в интернет.
  • Возможные задержки: дополнительный «прыжок» через прокси может замедлить работу, особенно при большой нагрузке.
  • Приватность сотрудников: логирование всей активности может вызывать вопросы по этике.
  • Стоимость и сложность настройки: особенно для крупных организаций.
Вопрос 10. Опиши роль избыточности (redundancy) при проектировании DNS-серверов. 4 балла

Что писать:

  • Несколько серверов с одинаковыми данными: если один сервер упал, другие продолжают отвечать на запросы — нет единой точки отказа.
  • Географическая распределённость: серверы стоят в разных регионах мира, что снижает задержку (latency) для пользователей и обеспечивает живучесть при катастрофах или сетевых сбоях.
  • Балансировка нагрузки: запросы распределяются между серверами, чтобы ни один не оказался перегружен. Это сохраняет производительность даже при росте трафика.
  • Особо устойчивы корневые серверы: их множество географически распределённых копий защищает от глобальных сбоев DNS.

🤝 A2.2.3 — Клиент-сервер vs Peer-to-Peer (P2P)

Это два фундаментально разных подхода к тому, как устройства общаются в сети.

👑 Клиент-сервер (Client-Server)

Один (или несколько) серверов хранят, обрабатывают и предоставляют ресурсы; клиенты запрашивают эти ресурсы. Модель построена по принципу запрос/ответ (request/response). Это централизованная архитектура — серверы выступают в роли централизованных точек доступа.

Пример: ты заходишь в браузер, набираешь youtube.com. Твой браузер — клиент. Сервер YouTube — сервер. Браузер шлёт HTTP-запрос → сервер находит видео → возвращает данные.

🤝 Peer-to-Peer (P2P, пиринговая сеть)

Децентрализованная модель: каждый участник (peer) имеет равные возможности и обязанности. Нет выделенного сервера — каждый узел может быть и клиентом, и сервером одновременно, обмениваясь ресурсами напрямую с другими peer'ами.

Пример: BitTorrent — файлы скачиваются маленькими кусочками сразу от десятков других пользователей; Bitcoin — блокчейн распределён по тысячам узлов; Skype (исторически) — звонки шли напрямую между пользователями.

Прямое сравнение

КритерийКлиент-серверPeer-to-Peer (P2P)
Контроль ✅ Централизованный. Легко применять политики, управлять ресурсами, обеспечивать целостность данных. ⚠️ Распределённый. Нет единой власти — устойчивее, но сложнее отслеживать проблемы и собирать метрики.
Масштабируемость Зависит от мощности сервера. Расширение = апгрейд серверов или добавление новых — дорого. ✅ Каждый новый peer добавляет ресурсы в общий пул. Масштабирование почти бесплатное.
Безопасность ✅ Централизованные меры безопасности проще администрировать. Но серверы — лакомая цель для атак. ⚠️ Анонимность может быть плюсом, но и минусом: труднее идентифицировать и пресекать вредоносное ПО.
Эффективность Серверы можно оптимизировать под конкретные задачи. Но при высокой нагрузке = bottleneck. Ресурсы распределены между peer'ами — снижает нагрузку на отдельные узлы. Но возможен оверхед при плохом управлении.
Надёжность ⚠️ Сервер = единая точка отказа. Если он упал — сервис недоступен всем. ✅ Сеть продолжает работать, даже если некоторые peer'ы отключились. Но peer'ы могут уходить/приходить — это влияет на доступность данных.

Реальные применения

🌐 Примеры клиент-сервер

  • Веб-сайты: браузер (клиент) запрашивает данные у веб-сервера. Преимущества — централизованное управление, единая версия контента; недостаток — если сервер ляжет, сайт недоступен всем.
  • Email с MS Exchange: почтовые клиенты (Outlook) обращаются к центральному почтовому серверу. Плюсы — централизованный контроль политик безопасности, резервное копирование, архивирование. Минусы — единая точка отказа, высокие затраты на обслуживание.
  • Онлайн-банкинг: банковское приложение → центральные серверы банка обрабатывают транзакции. Плюсы — безопасность и целостность данных. Минусы — серверы банка — лакомая цель для кибератак, и любой даунтайм аффектит всех клиентов.

🔗 Примеры P2P

  • BitTorrent: файлы скачиваются напрямую между пользователями, без центрального сервера. Плюсы — нет нужды в мощных серверах; быстрее для больших файлов (одновременная передача кусочков от многих peer'ов). Минусы — меньше контроля над безопасностью, скорость зависит от числа peer'ов.
  • Skype (классический): использовал P2P для прямого установления голосовых соединений. Плюсы — масштабируется без огромных вложений в инфраструктуру, лучшее качество за счёт прямой маршрутизации. Минусы — нестабильность peer'ов, сложно проводить централизованные политики безопасности.
  • Bitcoin и блокчейн: P2P-сеть, где каждый узел хранит копию журнала транзакций. Плюсы — децентрализация повышает безопасность, нет единой точки отказа, защита от подделок. Минусы — большие энергозатраты на консенсус (proof-of-work), транзакции могут быть медленнее, чем в централизованных системах.

Тренировочные вопросы — A2.2.3 Модели сетей

Вопрос 11. Сравни модели клиент-сервер и peer-to-peer по двум критериям. 4 балла

Что писать (любые 2 пары × 2 балла):

  • Контроль: в клиент-серверной модели контроль централизован на сервере → легко применять политики и управлять ресурсами. В P2P — контроль распределён между peer'ами, нет единого центра.
  • Надёжность: в клиент-серверной модели сервер — единая точка отказа (сервер упал → сервис недоступен). В P2P сеть продолжает работать даже при отказе нескольких peer'ов.
  • Масштабируемость: клиент-сервер требует апгрейда сервера при росте нагрузки (дорого). В P2P добавление нового peer'а само по себе увеличивает ресурсы сети.
  • Безопасность: в клиент-серверной модели меры безопасности применяются централизованно, но серверы — мишень для атак. В P2P — анонимность может быть плюсом, но затрудняет борьбу с вредоносным ПО.
Вопрос 12. Опиши один реальный пример клиент-серверной модели и один пример P2P, объяснив, почему именно эта модель здесь подходит. 4 балла

Что писать:

  • Клиент-сервер: онлайн-банкинг. Банковское приложение (клиент) обращается к серверам банка. Это правильная модель, потому что нужна централизованная безопасность, контроль транзакций и гарантия консистентности баланса. Распределённый P2P-подход здесь рискован — нельзя позволить, чтобы транзакции зависели от частных peer'ов.
  • P2P: BitTorrent. Файлы передаются напрямую между пользователями. Это правильная модель, потому что нет нужды в дорогих центральных серверах, а скачивание ускоряется при росте числа peer'ов (каждый отдаёт куски). Для централизованной модели потребовалась бы огромная пропускная способность сервера-источника.
Вопрос 13. Объясни, почему клиент-серверная модель имеет «единую точку отказа», и как P2P этого избегает. 3 балла

Что писать:

  • В клиент-серверной модели все клиенты обращаются к одному (или нескольким) центральным серверам. Если сервер упал из-за поломки, кибератаки или планового обслуживания, все клиенты теряют доступ к услуге.
  • В P2P нет выделенного сервера: каждый peer — одновременно и клиент, и сервер. Ресурсы распределены по сети.
  • Если один (или даже многие) peer'ы отключаются, оставшиеся peer'ы продолжают предоставлять данные и сервис. Поэтому P2P-сеть устойчивее к точечным сбоям.

Совет: в реальности крупные клиент-серверные системы используют кластеры серверов и репликацию, чтобы снизить риск, но архитектурно «единая точка отказа» — характеристика модели.

🧩 A2.2.4 — Сегментация сетей

Сегментация сети (network segmentation) — это разделение компьютерной сети на меньшие, отдельные сегменты или подсети. Это делают ради безопасности, производительности и удобства управления. Грубая аналогия: вместо одной большой комнаты с 200 людьми, шумящими одновременно, ты строишь 10 комнат по 20 человек — внутри каждой удобно работать, между комнатами трафик контролируется.

Зачем сегментировать сеть?

🔐 Безопасность
  • Только разрешённый трафик ходит между сегментами.
  • Пример: гостевой Wi-Fi в банке отделён от сети с клиентскими счетами — устройство гостя физически/логически не может попасть в платёжную сеть.
  • Если злоумышленник пробил один сегмент, он не получает доступ к остальным.
  • Можно применять разные политики безопасности к разным сегментам.
⚡ Производительность
  • Меньше широковещательного (broadcast) трафика в каждом сегменте → меньше «шума».
  • Тяжёлый трафик (видеостриминг, бэкапы) изолирован — не мешает остальной сети.
  • Пример: 50 студентов, стримящих видео в одном сегменте, не повлияют на другие сегменты сети.
📈 Масштабируемость
  • Новые сегменты добавляются без нарушения работы существующих.
  • Контролируемое расширение сети.
🛠️ Управляемость
  • Админам проще мониторить и управлять трафиком внутри сегментов.
  • Быстрее находят и устраняют проблемы.
  • Можно апгрейдить отдельные сегменты без перестройки всей сети.

Методы сегментации

🔢 A. Subnetting (Подсети)

Разделение большой IP-сети на меньшие логические подсети (subnets). Достигается манипуляцией маской подсети (subnet mask).

Ключевые понятия:

  • Маска подсети определяет, какая часть IP-адреса — «сеть», а какая — «хост». Например, при маске 255.255.255.0 первые 3 октета IP — это сеть, последний — хост.
  • Сетевой адрес определяется операцией побитового AND между IP и маской.
  • Устройства с одинаковой сетевой частью могут общаться напрямую; устройствам из разных подсетей нужен маршрутизатор.
  • CIDR-нотация (например, 192.168.1.0/24) — компактная запись: число после слэша = сколько бит маски равны 1.

Пример (Worked Example 2 из учебника):
IP: 192.168.1.10 → бинарно 11000000.10101000.00000001.00001010
Маска: 255.255.255.0 → бинарно 11111111.11111111.11111111.00000000
Побитовое AND даёт: 192.168.1.0 — это сетевой адрес.

Плюсы
  • Эффективнее использует IP-адресное пространство.
  • Уменьшает размер таблиц маршрутизации — роутеры хранят маршруты к подсетям, а не к каждому IP.
  • Изолирует широковещательный трафик внутри подсети.
  • Можно применять ACL (Access Control Lists) на границах подсетей для безопасности.
Минусы
  • Сложнее проектировать, нужно понимание двоичной арифметики.
  • Для общения между подсетями обязательно нужен маршрутизатор.

ACL (Access Control List): правила «кто к чему имеет доступ» — например, «компьютеры из подсети сотрудников могут читать файлы на файловом сервере, но компьютеры гостей — нет».

🏷️ B. VLAN (Virtual Local Area Network)

Виртуальная локальная сеть позволяет сегментировать физическую сеть на несколько логических сетей. Устройства можно группировать вместе, даже если они подключены к разным коммутаторам или находятся в разных физических местах.

Ключевые свойства:

  • Группировка по функции/отделу/приложению, а не по физическому расположению.
  • VLAN ограничивают широковещательные домены — широковещательный трафик идёт только тем, кому он нужен.
  • Изолированный трафик между VLAN → лучшая безопасность. Для связи между VLAN нужен inter-VLAN routing (маршрутизатор или L3-коммутатор), и его можно контролировать политиками.
  • Логическое разделение без физического: не нужно прокладывать отдельные кабели и покупать отдельные коммутаторы для каждой группы.

Пример из учебника: небольшая компания, 24-портовый коммутатор. Админ назначает порты 1-10 → VLAN 10 («Бухгалтерия»), порты 11-20 → VLAN 20 («Продажи»). Компьютеры из VLAN 10 могут общаться только между собой, не с VLAN 20. Уменьшается ненужный широковещательный трафик, повышается безопасность, можно применять разные QoS-политики.

Плюсы
  • Безопасность: чувствительные данные отделены от остального трафика.
  • Гибкость: группировка не зависит от физического подключения.
  • Производительность: ограничение широковещательного домена.
  • Дешевле, чем покупать отдельные коммутаторы для каждой группы.
Минусы
  • Требует управляемых коммутаторов (managed switches) — они дороже простых.
  • Сложнее настройка и поддержка, особенно в больших сетях.
  • Неправильная конфигурация может создать «дыру» в безопасности или сломать связь.

Subnetting + VLAN = вместе

В реальности subnetting и VLAN часто работают вместе. VLAN сегментирует на уровне коммутатора (L2), а subnetting — на уровне IP (L3). Например: каждая VLAN получает свою подсеть, и inter-VLAN routing между ними контролируется ACL'ами. Это даёт двойную защиту и оптимизацию.

Тренировочные вопросы — A2.2.4 Сегментация

Вопрос 14. Объясни два преимущества сегментации сети. 4 балла

Что писать (любые 2 × 2 балла):

  • Безопасность: только разрешённый трафик может пересекать границы сегментов. Если злоумышленник пробил один сегмент (например, гостевой Wi-Fi), он не получает доступ к чувствительным сегментам (например, бухгалтерии). Можно применять разные политики безопасности к разным сегментам.
  • Производительность: сегментация ограничивает широковещательный трафик внутри сегмента и распределяет нагрузку. Если 50 студентов стримят видео в одном сегменте, это не влияет на работу других сегментов.
  • Управляемость: админам проще мониторить и устранять проблемы в меньших сегментах. Можно апгрейдить части сети без её полной перестройки.
  • Масштабируемость: новые сегменты добавляются, не нарушая работу существующих — сеть может расти контролируемо.
Вопрос 15. Применяя маску подсети 255.255.255.0 к IP-адресу 192.168.5.42, определи сетевую часть и хостовую часть. 3 балла

Что писать:

  • Маска 255.255.255.0 означает, что первые 3 октета — сетевая часть, последний октет — хостовая часть.
  • Сетевая часть: 192.168.5 (или сетевой адрес 192.168.5.0).
  • Хостовая часть: 42 (то есть конкретное устройство в этой сети).
  • Бонус: любые устройства с IP в диапазоне 192.168.5.1–192.168.5.254 находятся в той же подсети и могут общаться напрямую без маршрутизатора.
Вопрос 16. Объясни, чем VLAN отличается от обычной подсети, и какие преимущества даёт использование VLAN. 4 балла

Что писать:

  • Subnetting разделяет сеть на уровне IP-адресов (L3): с помощью маски подсети группа IP-адресов считается одной подсетью. Устройствам разных подсетей нужен маршрутизатор для общения.
  • VLAN разделяет сеть на уровне коммутатора (L2): устройства группируются логически, даже если физически подключены к разным коммутаторам или находятся в разных местах.
  • Преимущества VLAN:
    • Группировка по функции/отделу, а не по физическому расположению — удобно для крупных офисов.
    • Лучшая безопасность: трафик между VLAN изолирован, для связи нужен маршрутизатор или L3-коммутатор с политиками доступа.
    • Меньше широковещательного трафика → лучше производительность.
    • Можно сегментировать сеть, не прокладывая новые кабели и не покупая отдельные коммутаторы.
  • На практике VLAN и subnetting часто работают вместе — каждый VLAN получает свою подсеть.
Вопрос 17. Какие меры безопасности можно реализовать с помощью сегментации в банке, где есть гостевой Wi-Fi и сеть с клиентскими счетами? 4 балла

Что писать:

  • Изоляция гостевого Wi-Fi от внутренней сети: устройства гостей физически/логически не могут попасть в сеть с клиентскими счетами. Это разные VLAN/подсети.
  • Применение ACL (Access Control Lists) на маршрутизаторе или firewall: разрешён трафик только из конкретных «доверенных» подсетей в платёжную систему.
  • Минимизация поверхности атаки: если злоумышленник захватил устройство в гостевом сегменте, он не получает доступ к серверам с банковскими данными.
  • Разные политики безопасности по сегментам: в платёжном сегменте могут быть строгие правила (шифрование, аудит, многофакторная аутентификация), в гостевом — упрощённые.
  • Контролируемый мониторинг: подозрительная активность внутри чувствительных сегментов фиксируется проще, потому что обычно там малый и предсказуемый трафик.

🔬 Практические задания

🔍 Индивидуальное: спроектируй сегментированную сеть для школы

Цель: применить понятия сегментации, VLAN и подсетей к реальному сценарию.

Сценарий: школа на 600 учеников и 50 учителей. Нужно: интернет для всех; общая локальная сеть для учителей с доступом к файловому серверу; отдельный гостевой Wi-Fi для родителей и посетителей; IoT-устройства (камеры, умные доски); компьютерный класс на 30 машин.

Инструкции:

  1. Определи, сколько сегментов (VLAN) нужно. Минимум 4: учителя, ученики, гостевой Wi-Fi, IoT. Возможно — отдельный для серверной.
  2. Для каждого сегмента распиши, какой трафик разрешён в и из него (например, «гостевой → только интернет, нельзя во внутреннюю сеть»).
  3. Выбери подходящие подсети (например, 192.168.10.0/24 для учителей, 192.168.20.0/24 для учеников, и т.д.).
  4. Объясни, почему такое разделение даёт лучшую безопасность и производительность, чем одна плоская сеть.

👥 Групповое: дебаты «Клиент-сервер vs P2P»

Цель: через дискуссию глубже понять плюсы и минусы каждой модели.

Инструкции (группы по 4):

  1. Разделитесь на 2 команды. Команда A защищает клиент-сервер, команда B — P2P.
  2. Каждая команда готовит аргументы на тему: «Для какого приложения ИИ-чатбот подходит лучше — клиент-сервер или P2P?»
  3. Учитывайте: скорость, безопасность, надёжность, стоимость, простоту обновлений модели.
  4. Поменяйтесь ролями: теперь A защищает BitTorrent-альтернативу, B — централизованный стриминговый сервис.
  5. Запишите общий вывод: какие признаки задачи намекают, что нужно идти в клиент-сервер, а какие — в P2P.

🏠 Домашнее: топология твоей сети дома

Цель: связать абстрактные знания с реальностью.

  1. Опиши, какая топология у тебя дома. Скорее всего, физически — звезда вокруг роутера. Логически — возможно, гибрид (если есть Wi-Fi mesh-расширители).
  2. Сосчитай устройства: компьютеры, смартфоны, smart-TV, умные колонки, IoT (камеры, лампочки). Сколько их в одной сети?
  3. Спросишь: сегментирована ли твоя сеть? Большинство домашних роутеров позволяют создать «гостевой Wi-Fi» (отдельный VLAN). Если у тебя есть IoT — желательно вынести его в отдельную сеть (по соображениям безопасности).
  4. Опиши, какие сервисы ты используешь и какая у них модель: Google Drive — клиент-сервер; µTorrent — P2P; WhatsApp-звонки — гибрид (стартует через сервер Meta, далее может быть P2P).

🎯 Что важно запомнить

  • Топология описывает форму сети. Для экзамена сравнивай по 5 осям: надёжность, скорость, масштабируемость, поведение при отказе, стоимость.
  • Звезда — самая популярная (дом, офис). Mesh — самая надёжная (нет единой точки отказа). Гибрид — комбинирует лучшее.
  • Различай физическую топологию (где лежат провода) и логическую (как ходят данные).
  • Серверы (HL) — это специализированные «работники» сети. Запомни 8 типов: DNS, DHCP, file, mail, proxy, web, database, application. Знай 1–2 основных протокола у каждого.
  • Клиент-сервер — централизованно, легко управлять, но единая точка отказа. P2P — децентрализованно, устойчиво, дешёво масштабируется, но сложнее контролировать.
  • Сегментация = лучшая безопасность + лучшая производительность + лучшая управляемость.
  • Subnetting делит сеть на уровне IP (через маску подсети); VLAN — на уровне коммутатора (логически, независимо от физики). Часто работают вместе.
  • Помни про DORA-процесс DHCP: Discovery → Offer → Request → Ack.
  • Маска подсети 255.255.255.0 = /24 в CIDR. Побитовый AND IP-адреса и маски = сетевой адрес.
  • На экзамене вопрос «опиши преимущества и недостатки X» — всегда даёшь 2 плюса + 2 минуса минимум (это шаблон для 4 баллов).
