Основы баз данных

Эта тема (A3.1) — про то, что такое база данных (database), зачем она нужна, и почему «просто файл» — это плохо, когда данных становится много. Ты узнаешь, что такое СУБД (DBMS), какие бывают типы баз, как устроены таблицы, ключи и связи, и почему целостность данных (data integrity) — главное, ради чего вообще строят базы. Каждый день ты пользуешься десятками баз, даже не замечая этого — в магазине, в банке, в школьном дневнике, в Instagram.

1. Что такое база данных и зачем она нужна

База данных (database) — это систематически организованное хранилище данных. Данные хранятся логично, обычно в нескольких связанных таблицах, чтобы их можно было быстро искать, обновлять, защищать и совместно использовать. Ты постоянно взаимодействуешь с базами данных, часто этого не осознавая.

💡 Представь библиотеку: книги — это записи (rows), полки с разной информацией о каждой книге (автор, название, год) — это поля (columns), а сама полка с однотипными карточками — это таблица (entity). Каталог, в котором всё это устроено и можно быстро найти любую книгу, — это и есть база данных.

Где ты уже встречался с базами данных

  • Розничная торговля (retail): в онлайн-магазине база хранит товары, покупателей и поставщиков; пересчитывает остатки на складе после каждого заказа.
  • Социальные сети: твой профиль, посты, лайки, переписки — всё это записи в базе данных Instagram/TikTok/VK.
  • Медицинские карты (medical records): история болезней, назначения, результаты анализов, информация о врачах и кабинетах хранятся в больничной БД.
  • Банковские системы: счета, транзакции, балансы — миллиарды записей, и каждая должна быть точной до копейки.
  • Игры (MMORPG): в играх вроде Lost Ark база хранит твои достижения, инвентарь, чаты, друзей. Когда ты сохраняешь игру — твой статус записывается в БД, чтобы при возврате всё было на месте.

2. Плоский файл vs. база данных: зачем вообще нужны БД

Самый простой способ хранить данные — это плоский файл (flat file): одна большая таблица с одной «сущностью» (entity = table). Звучит удобно, но на практике быстро ломается. Возьмём пример из учебника — ветеринарная клиника, где всё свалено в одну таблицу: владелец, питомец, врач, лечение.

2032Theo Naidoo45 Rue Martignac2393KaiDog4400Rachel17/08/25Cast Removal200

Видишь проблему? Имя Alison Bachman и её адрес повторяются для каждого её питомца. Имя врача Rachel и её адрес — тоже. Это называется избыточность данных (data redundancy), и из-за неё возникают три классические аномалии (anomalies).

Три аномалии плоского файла

➕ Аномалия вставки (Insert anomaly)

Нельзя добавить нового врача, пока он не пролечил хотя бы одного питомца. Получается абсурд: вет должен сначала кого-то полечить, чтобы попасть в базу. Невозможно вставить независимую сущность.

➖ Аномалия удаления (Deletion anomaly)

Нельзя удалить лечение, не потеряв заодно информацию о враче, питомце или владельце. Например, если ты удалишь все записи про вет-врача Lucy, ты потеряешь и данные про кота Seb.

✏️ Аномалия обновления (Update anomaly)

Если владелец Alison сменит адрес и ты обновишь только одну строку — в базе будут две разных версии её адреса, и непонятно, какая правильная. Это называется data inconsistency (несогласованность данных).

Почему вообще не хранить всё в одной таблице

  • Огромный объём. Повторяющиеся данные занимают много места — и место кончается быстро.
  • Сложно масштабировать. Чем больше данных, тем чаще возникают аномалии.
  • Неточные расчёты. Любые SQL-запросы и аналитика дадут неправильные результаты, если базовые данные противоречат друг другу.
  • Нет совместного доступа. Один файл — один пользователь за раз; никакой нормальной работы команды.

Решение — разбить плоский файл на несколько связанных таблиц. Это называется нормализация (normalization), а получившаяся структура — реляционная база данных (relational database).

3. СУБД (DBMS) — что управляет базой

Сама база данных — это просто структурированные файлы на диске. Чтобы их создавать, изменять, искать в них и защищать, нужен специальный софт — система управления базами данных (Database Management System, DBMS). СУБД — это посредник между пользователем (или приложением) и реальными данными.

⚙️ Что делает СУБД

Назначение
Управляет хранением, поиском, обновлением и безопасностью данных в базе.
Основные функции
  • Хранение данных на диске в эффективном формате (индексы, страницы, журналы).
  • Выполнение запросов на языке SQL — SELECT, INSERT, UPDATE, DELETE.
  • Контроль одновременного доступа (concurrency control) — несколько пользователей могут работать без конфликтов.
  • Безопасность и права доступа — кто что может видеть и менять.
  • Резервное копирование (backup) и восстановление после сбоев.
  • Поддержка целостности (integrity) — данные не противоречат правилам базы.
Популярные примеры
  • MySQL — самая популярная open-source реляционная СУБД; стоит за миллионами сайтов.
  • PostgreSQL — мощная open-source СУБД, любима за расширяемость и стандартизацию.
  • SQLite — лёгкая встраиваемая СУБД; работает в одном файле, без отдельного сервера (используется в браузерах, телефонах, нашей SQL-песочнице).
  • Oracle Database — мощная корпоративная СУБД, часто в банках и крупных компаниях.
  • Microsoft SQL Server — корпоративная СУБД от Microsoft.
  • MongoDB — пример NoSQL СУБД (документная).
💡 Аналогия: база данных — это книжный склад. СУБД — это менеджер склада: он знает, где что лежит, выдаёт книги по запросу, следит, чтобы никто не унёс лишнего, и ведёт журнал, кто что взял и вернул.

4. Типы баз данных

Не все базы устроены одинаково. Реляционная — самая популярная, но не единственная. Вот основные модели, с которыми ты можешь встретиться:

Тип Как хранит данные Где применяют
Реляционная (Relational) Таблицы со строками и столбцами, связанные через ключи. Язык запросов — SQL. Бизнес, школа, банк, e-commerce — везде, где данные структурированы.
NoSQL (документная, ключ-значение, графовая, колоночная) Гибкие схемы: документы (JSON), пары ключ-значение, графы или колоночные форматы. Большие данные, реальное время, аналитика, соцсети (MongoDB, Redis, Neo4j, Cassandra).
Объектно-ориентированная (Object-oriented) Хранит объекты целиком (как в Java/Python), с их полями и методами. Сложные данные с поведением — CAD-системы, мультимедиа, научные симуляции.
Иерархическая (Hierarchical) Данные в виде дерева: «родитель → дети». Каждая запись имеет одного родителя. Файловые системы, организационные структуры, старые мейнфреймы (IBM IMS).
Сетевая (Network) Как иерархическая, но запись может иметь несколько родителей. Граф связей. Сложные взаимосвязи (старые ERP), где иерархия слишком ограниченна.

Когда выбирать что

🗂️ Реляционная — выбор по умолчанию

Если данные чётко структурированы (имеют одинаковые поля у всех записей) и важна целостность — это реляционная база. Это 90% бизнес-задач: интернет-магазин, школьный журнал, банк.

📄 NoSQL — для гибкости и масштаба

Если структура данных сильно меняется (например, каждый пользователь хранит свои настройки), либо данных очень много (big data) — реляционная плохо тянет. Здесь приходит NoSQL: MongoDB хранит JSON-документы, Redis — пары ключ-значение для кэша, Neo4j — графы для социальных сетей.

🌲 Иерархическая и сетевая — устаревшие

Сейчас редко встречаются как основная БД, но идея жива в файловых системах, организационных деревьях и XML/JSON.

Ключевые термины

📋 Сущность / Таблица (Entity / Table / Relation)

Что это
Таблица в базе данных. Каждая сущность хранит данные об одном типе объектов: владельцы, питомцы, врачи, лечения.
Пример
Сущность PATIENT хранит данные обо всех животных в клинике.
Тип
Структура

📝 Запись / Кортеж (Row / Tuple / Record)

Что это
Одна строка в таблице. Содержит данные об одном конкретном экземпляре сущности.
Пример
Строка (1011, Oskar, Dog, 1029) — это запись об одном питомце Оскаре.
Тип
Структура

📐 Поле / Атрибут (Column / Attribute / Field)

Что это
Столбец в таблице. Представляет характеристику данных. У каждой записи в таблице один и тот же набор атрибутов.
Пример
В PATIENT атрибуты: Patient_ID, Name, Type, Owner_ID.
Тип
Структура

🔑 Первичный ключ (Primary Key, PK)

Что это
Уникальный идентификатор каждой строки в таблице. Не повторяется и не бывает пустым.
Зачем
В таблице может быть несколько собак по имени Oskar. Но Patient_ID = 1011 — только один. Это позволяет однозначно сослаться на запись.
Пример
Patient_ID в таблице PATIENT, Owner_ID в таблице OWNER.

🔗 Внешний ключ (Foreign Key, FK)

Что это
Поле в одной таблице, которое ссылается на первичный ключ другой таблицы. Так создаются связи между таблицами.
Зачем
В PATIENT хранится Owner_ID — это FK, который указывает, какому владельцу из OWNER принадлежит питомец. Так мы избегаем дублирования имени и адреса хозяина.
Пример
В PATIENT: Owner_ID = 1029 → указывает на запись Alison Bachman в OWNER.

🎯 Кандидатный ключ (Candidate Key)

Что это
Любое поле (или комбинация полей), которое могло бы стать первичным ключом — то есть однозначно идентифицирует запись.
Зачем
В таблице может быть несколько уникальных полей: например, в таблице студентов уникальны и Student_ID, и email. Оба — кандидатные ключи. Первичным выбирают только один из них.

🧩 Составной ключ (Composite Key)

Что это
Первичный ключ, состоящий из двух или более полей — потому что ни одно поле по отдельности не уникально.
Пример
В таблице TREATMENT поле Patient_ID не уникально (у одного питомца много лечений), но Patient_ID + Date уже уникально — питомец не получает два лечения в один день.

🪢 Конкатенированный ключ (Concatenated Key)

Что это
Ключ, собранный из фрагментов нескольких атрибутов, чтобы получить уникальное значение.
Пример
В таблице людей: первые 3 буквы имени + первые 3 буквы фамилии + дата рождения = почти всегда уникально.

Связи между таблицами (Relationships)

Если у таблиц нет связей, данные «лежат в одиночестве» и пользы от них немного. Связь обозначает, как записи в одной таблице соотносятся с записями в другой.

Кардинальность (Cardinality) — тип связи

1️⃣ Один-к-одному (One-to-One, 1:1)

Одной записи в одной таблице соответствует ровно одна запись в другой. Пример: один питомец имеет ровно одного владельца (в нашей упрощённой модели).

1️⃣Ⓜ️ Один-ко-многим (One-to-Many, 1:N)

Одной записи в первой таблице соответствует много записей во второй. Пример: один вет выполняет много лечений; один владелец имеет много питомцев.

Ⓜ️Ⓜ️ Многие-ко-многим (Many-to-Many, M:N)

Много записей в первой связаны с многими во второй. Пример: много студентов записаны на много курсов. Реализуется через промежуточную таблицу-связку.

Модальность (Modality) — обязательность связи

❓ Опциональная связь (Optional)

Связь может быть, а может и не быть. Пример: «Актёр — ТВ-шоу». Актёр не обязан сниматься в сериале (может играть только в театре).

❗ Обязательная связь (Mandatory)

Связь должна быть. Пример: «Чек — Товары». Чтобы существовал чек, в нём обязан быть хотя бы один товар.

6. Целостность данных (Data Integrity)

Целостность данных — это набор правил, гарантирующих, что данные в базе точные, согласованные и достоверные. Без неё база — мусорка с противоречиями. Различают три классических типа.

🆔 Целостность сущности (Entity Integrity)

Правило
Каждая строка должна иметь уникальный, непустой первичный ключ. Не может быть двух строк с одинаковым PK; PK не может быть NULL.
Зачем
Гарантирует, что мы всегда можем однозначно сослаться на запись. Без этого нельзя сказать «удали строку с Patient_ID = 1011», если таких строк несколько.
Пример нарушения
Два пациента с Patient_ID = 1011 — запутаемся, какого Оскара лечили.

🔗 Ссылочная целостность (Referential Integrity)

Правило
Внешний ключ должен указывать на существующий первичный ключ в связанной таблице (или быть NULL, если связь опциональна).
Зачем
Не должно быть «висящих» ссылок: питомца, чей Owner_ID указывает на несуществующего владельца.
Пример нарушения
В PATIENT записан Owner_ID = 9999, но в OWNER нет такого ID. Это orphan record — «сирота».

📐 Доменная целостность (Domain Integrity)

Правило
Значения в каждом столбце должны соответствовать заданному типу и допустимому диапазону. Это включает тип данных (INT, VARCHAR, DATE), ограничения (NOT NULL, CHECK), формат.
Зачем
Защищает от «мусора»: возраст не может быть отрицательным, цена не может быть «hello», дата рождения не в 3025 году.
Пример нарушения
В поле Cost (число) кто-то записал «бесплатно». В поле Type (Dog/Cat/Hamster) — «бутылка».

Бонус: ACID — четыре буквы надёжных транзакций

Реляционные СУБД соблюдают ACID-свойства для транзакций (нескольких операций, выполняемых как единое целое):

  • A — Atomicity (атомарность): либо вся транзакция выполняется, либо ни одна её часть. Пример: при банковском переводе либо обе операции (списать у А, зачислить Б) проходят, либо ни одна.
  • C — Consistency (согласованность): транзакция переводит базу из одного валидного состояния в другое.
  • I — Isolation (изолированность): одновременные транзакции не мешают друг другу.
  • D — Durability (устойчивость): после успешного коммита данные сохраняются даже при сбое сервера.

7. Кто работает с базой данных — роли

👨‍💼 Администратор БД (DBA — Database Administrator)

Кто это
Технический специалист, отвечающий за работу и здоровье базы данных в продакшене.
Что делает
  • Устанавливает и настраивает СУБД.
  • Контролирует безопасность: права доступа, шифрование, аудит.
  • Делает резервные копии (backups) и проверяет их восстановление.
  • Оптимизирует производительность: индексы, запросы, нагрузка.
  • Следит за целостностью и решает проблемы при сбоях.
Когда нужен
В банках, больницах, корпорациях — везде, где база критически важна 24/7.

🎨 Проектировщик БД (Database Designer)

Кто это
Специалист, который проектирует структуру базы данных до того, как она будет реализована.
Что делает
  • Анализирует требования бизнеса: какие данные хранить, какие запросы будут.
  • Строит ER-диаграммы (Entity-Relationship Diagrams).
  • Определяет сущности, атрибуты, ключи и связи.
  • Выполняет нормализацию, чтобы минимизировать аномалии.
  • Подбирает типы данных и ограничения.

👤 Конечный пользователь (End User)

Кто это
Тот, кто пользуется базой через приложение — не пишет SQL напрямую (обычно).
Что делает
  • Заполняет формы (регистрация, заказ, отправка сообщения).
  • Ищет данные через поисковики и фильтры.
  • Видит отчёты и графики, основанные на запросах к БД.
Пример
Кассир в магазине, который пробивает товар; пациент в личном кабинете больницы; ученик в школьном дневнике; ты, когда листаешь Instagram.

8. Плюсы и минусы реляционной БД

Преимущества (Benefits)

ПреимуществоЧто это значит
Поддержка сообществаОгромное community: документация, форумы, готовые решения. MySQL и PostgreSQL знают миллионы разработчиков.
Контроль одновременного доступаНесколько пользователей могут одновременно читать и менять данные без ошибок (благодаря ACID-свойствам).
Согласованность данных (Data consistency)Меньше повторов = меньше противоречий. Адрес владельца хранится один раз и везде «правильный».
Целостность данных (Data integrity)Меньше повторов = меньше ошибок. PK, FK и ограничения защищают от мусора.
Простой поиск (Data retrieval)SQL — мощный язык запросов. Один запрос может перебрать миллион записей за секунды.
Сокращение дублирования (Reduced duplication)Каждая часть информации хранится один раз — экономия места.
Сокращение избыточности (Reduced redundancy)За счёт связей между таблицами одни и те же данные не «размножаются» по базе.
Надёжные транзакцииРеляционные СУБД проверяют каждую транзакцию (ACID), чтобы переводы денег и заказы не «терялись».
МасштабируемостьЛучше тянет рост, чем плоский файл (но хуже, чем NoSQL для big data).
БезопасностьВстроенные права доступа: можно дать одному пользователю только чтение, другому — изменение, третьему — полный доступ.

Недостатки (Limitations)

НедостатокВ чём проблема
Плохо тянет big dataДля аналитики на терабайтных объёмах данных реляционные модели не лучший выбор. Лучше колоночные БД или NoSQL.
Сложность проектированияСпроектировать правильно нормализованную схему с десятками таблиц — нетривиальная задача.
Иерархические данные неудобныДеревья (например, комментарии с ответами на ответы) хранятся плохо, обход медленный.
Жёсткая схема (Rigid schema)Типы и связи зафиксированы. Если завтра нужно добавить новое поле всем пользователям — это миграция, а не «само».
Object-relational impedance mismatchВ коде программы (Java, Python) данные — это объекты; в БД — таблицы. Перевод между ними («ORM») всегда не на 100% бесшовный.
Неструктурированные данныеТекст письма, фотография, JSON произвольной формы — плохо ложатся в табличную модель.
Вопрос 4. Крупная современная больница модернизирует свою IT-инфраструктуру. Они планируют использовать объектно-ориентированный (OOP) язык программирования для нового ПО управления пациентами. Они обсуждают, стоит ли хранить все данные — структурированные данные о пациентах, изменчивые аудио-заметки врачей и разнообразные файлы рентгеновских снимков — в стандартной реляционной БД. Обсудите ограничения (limitations), с которыми столкнётся больница, если выберет реляционную БД в этом сценарии. 6 баллов

Что писать: До 6 баллов. Студенты должны явно связывать ограничения с контекстом больницы, чтобы получить полный балл.

Работа с неструктурированными данными (Unstructured Data Handling) — до 3 баллов:

  • Реляционные базы данных требуют жёсткой схемы (rigid schema) строк и столбцов для представления данных. [1]
  • Неструктурированные данные — изменчивые аудио-заметки и разные форматы рентгеновских снимков — не имеют общих полей или атрибутов. [1]
  • Следовательно, практически невозможно спроектировать реляционную схему для эффективного хранения и поиска разнообразных мультимедийных файлов больницы. [1]

Object-Relational Impedance Mismatch — до 3 баллов:

  • Больница использует OOP-язык, в котором применяются концепции инкапсуляции (encapsulation), полиморфизма (polymorphism) и наследования (inheritance). [1]
  • Реляционные БД не могут напрямую отобразить или перевести все OOP-концепции (например, не существует способа перевести OOP-наследование в реляционно-табличную концепцию). [1]
  • Это «несовпадение» (mismatch) приведёт к неэффективности и потребует сложного обходного кода для взаимодействия программы больницы с базой данных. [1]

9. Реальные примеры баз данных

🛒 Онлайн-магазин (e-commerce)

Таблицы: Users (покупатели), Products (товары), Orders (заказы), OrderItems (позиции в заказе), Categories (категории).

Связи: один пользователь делает много заказов (1:N). Один заказ содержит много позиций (1:N). Каждая позиция ссылается на товар (N:1).

Целостность: нельзя оформить заказ от несуществующего пользователя (ссылочная целостность); нельзя поставить цену «hello» (доменная); каждый заказ имеет уникальный Order_ID (целостность сущности).

🏦 Банк

Таблицы: Accounts (счета), Customers (клиенты), Transactions (операции), Branches (отделения).

Здесь критически важны ACID-свойства: при переводе денег нужно атомарно списать с одного счёта и зачислить на другой — никаких «полусделок». Если сервер упал между операциями — транзакция откатывается целиком.

DBA в банке — это серьёзная работа: ошибка в БД = потерянные деньги клиентов = регулятивные штрафы.

🏥 Больница (медицинские записи)

Таблицы: Patients, Doctors, Appointments, Prescriptions, Diagnoses, Departments.

Сложные связи (M:N): пациент посещает разных врачей, врач принимает множество пациентов — через промежуточную таблицу Appointments.

Безопасность критична: только лечащий врач имеет доступ к карте пациента (роль-based access). Конфиденциальность защищена законом (HIPAA в США, GDPR в ЕС).

Вопрос 5. Обсудите одну этическую проблему или проблему конфиденциальности (privacy concern), которая возникает при централизации высокочувствительных данных сущностей (например, медицинских карт пациентов) в единой взаимосвязанной реляционной базе данных. 3 балла

Что писать: До 3 баллов за развёрнутое обсуждение этики/конфиденциальности.

  • За счёт централизации всех данных через связи первичных и внешних ключей, одно нарушение безопасности (security breach) раскрывает весь профиль пациента целиком. [1]
  • Поскольку данные крайне чувствительные (Personally Identifiable Information / медицинские истории), несанкционированный доступ может привести к краже личности (identity theft), дискриминации или серьёзному эмоциональному стрессу. [1]
  • Следовательно, у администраторов баз данных есть строгое этическое обязательство внедрять надёжные меры безопасности — такие как шифрование (encryption) и role-based access control, — чтобы защитить целостность данных и конфиденциальность сущностей, хранящихся в БД. [1]

10. Практические задания

Эти задания помогут проверить, что ты понимаешь A3.1 на уровне применения, а не только запоминания.

🔍 Индивидуально: нормализуй плоский файл (ATL — Thinking Skills)

Цель: Превратить плоскую таблицу в нормализованную реляционную модель.

Инструкции:

  1. Возьми таблицу из реального источника — например, расписание уроков в школе или меню в столовой.
  2. Найди повторяющиеся данные (имя учителя, название блюда, цены) и поставь их в отдельные таблицы.
  3. Определи первичные ключи для каждой таблицы.
  4. Установи внешние ключи, чтобы связать таблицы.
  5. Нарисуй ER-диаграмму и укажи кардинальность каждой связи (1:1, 1:N, M:N).
  6. Объясни, какие аномалии (insert, delete, update) ты устранил.

👥 В группе: спроектируй БД для библиотеки (ATL — Social & Communication)

Цель: Совместно спроектировать реляционную базу для школьной библиотеки.

Сценарий: Школьной библиотеке нужна БД для учёта книг, читателей и выдач. Каждая книга может быть выдана много раз; один читатель может одновременно держать несколько книг.

Инструкции (группы по 3-4):

  1. Определи сущности (минимум 3-4): Book, Reader, Loan, возможно Author.
  2. Перечисли атрибуты каждой сущности и выбери первичные ключи.
  3. Установи связи и определи кардинальность: один читатель → много выдач (1:N); одна книга → много выдач (1:N).
  4. Для каждой связи укажи модальность: обязательная или опциональная?
  5. Подумай, какие правила целостности данных вы будете применять: нельзя выдать несуществующую книгу, дата возврата не может быть раньше даты выдачи и т.д.
  6. Презентуйте решение классу: ER-диаграмма + 2-3 примера SQL-запросов, которые вы хотели бы поддерживать («показать все книги, которые читатель ещё не вернул»).

🏠 Домой: исследование (ATL — Research Skills)

Цель: Связать теорию A3.1 с реальной системой, которой ты пользуешься.

Инструкции:

  1. Выбери приложение, которым пользуешься регулярно: Telegram, Spotify, школьный электронный дневник, любимая игра.
  2. Перечисли минимум 5 сущностей, которые, по твоему мнению, есть в БД этого приложения.
  3. Определи тип связей между ними (1:1, 1:N, M:N) — например, в Spotify: User ↔ Playlist (1:N), Playlist ↔ Track (M:N).
  4. Подумай: какой тип БД (relational / NoSQL / гибридный) лучше всего подошёл бы и почему.
  5. Опиши одну проблему целостности, которая могла бы возникнуть в этой системе (например, что будет, если удалить аккаунт пользователя — что произойдёт с его плейлистами?).

Что важно запомнить

  • База данных = структурированное хранилище с правилами целостности. СУБД (DBMS) — софт, который этим управляет (MySQL, PostgreSQL, SQLite, Oracle).
  • Плоский файл приводит к трём аномалиям: insert, delete, update. Поэтому данные разбивают на таблицы — это нормализация.
  • Каждая таблица — сущность (entity). Строки — записи (tuples). Столбцы — атрибуты (attributes).
  • Первичный ключ (PK) — уникальный, непустой идентификатор. Внешний ключ (FK) — ссылка на PK другой таблицы. Так создаются связи.
  • Бывают также: candidate key (любой кандидат на PK), composite key (PK из нескольких полей), concatenated key (склейка из фрагментов).
  • Кардинальность: 1:1, 1:N, M:N. Модальность: обязательная или опциональная.
  • Три типа целостности: entity (уникальный PK), referential (FK указывает на существующий PK), domain (значения соответствуют типу и диапазону).
  • Роли: DBA эксплуатирует БД в проде; проектировщик создаёт схему; конечный пользователь работает через интерфейс.
  • Плюсы реляционных БД: целостность, согласованность, SQL, ACID, безопасность. Минусы: жёсткая схема, плохо с big data и неструктурированными данными — там лучше NoSQL.
  • На экзамене популярные вопросы: «опиши X аномалий», «объясни PK и FK», «приведи пример 1:N», «два преимущества реляционной БД». Всегда давай определение + пример.
