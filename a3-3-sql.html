SQL и программирование БД

Эта тема (A3.3) — про то, как реально разговаривать с базой данных. Язык, на котором мы говорим, называется SQL (Structured Query Language — «структурированный язык запросов»). С его помощью мы создаём таблицы, добавляем данные, ищем, обновляем, удаляем и связываем информацию из разных таблиц. Тут мы разберём четыре подсемейства SQL (DDL/DML/DCL/TCL), все ключевые операторы, объединения таблиц (JOIN), агрегатные функции, подзапросы, транзакции и свойства ACID.

1. Четыре категории SQL

СУБД (Система Управления Базами Данных, англ. Database Management System) использует четыре подсемейства языка SQL. Они отличаются задачей: одни строят структуру, другие работают с данными, третьи управляют доступом, четвёртые — целостностью транзакций. На экзамене любят спрашивать «к какой категории относится команда X?» — выучи эту таблицу.

DDL · Data Definition Language

Язык определения данных. Создаёт схему — таблицы, поля, ограничения.
  • CREATE — создать объект (таблицу, БД, индекс)
  • ALTER — изменить структуру таблицы
  • DROP — удалить объект целиком
  • TRUNCATE — удалить все записи (саму таблицу оставить)
  • RENAME — переименовать объект
  • COMMENT — добавить комментарий

DML · Data Manipulation Language

Язык манипуляции данными. Работает внутри таблиц.
  • SELECT — извлечь данные
  • INSERT — добавить запись
  • UPDATE — изменить запись
  • DELETE — удалить записи (структура остаётся)
  • CALL — вызвать процедуру SQL
  • LOCK TABLE — заблокировать таблицу (для параллельной работы)

DCL · Data Control Language

Язык управления доступом. Кто что может делать.
  • GRANT — выдать привилегию (право на действие)
  • REVOKE — отозвать привилегию

TCL · Transaction Control Language

Язык управления транзакциями. Сохраняем или откатываем.
  • COMMIT — закрепить (сохранить) изменения навсегда
  • ROLLBACK — откатить к последнему COMMIT
  • SAVEPOINT — поставить «закладку» для частичного отката
💡 Аналогия: DDL — это архитектор (строит здание). DML — жилец (двигает мебель внутри). DCL — охранник (выдаёт пропуска). TCL — нотариус (фиксирует или отменяет сделку).
Вопрос 1.1. К какой категории SQL относится команда UPDATE, а к какой GRANT? Обоснуй ответ. 2 балла
  • UPDATE относится к DML (Data Manipulation Language) — она меняет данные внутри существующей записи. Структура таблицы не трогается.
  • GRANT относится к DCL (Data Control Language) — она выдаёт права доступа пользователю (например, разрешает делать SELECT на таблицу). К данным напрямую не обращается.
Вопрос 1.2. Объясни разницу между DROP TABLE, DELETE FROM и TRUNCATE TABLE. 4 балла
  • DROP TABLE имя; (DDL) — удаляет таблицу целиком: и данные, и саму структуру. После этого таблицы просто не существует.
  • DELETE FROM имя WHERE ...; (DML) — удаляет отдельные строки, которые подходят под условие. Таблица и её колонки остаются. Можно откатить через ROLLBACK (если в транзакции).
  • TRUNCATE TABLE имя; (DDL) — удаляет все строки сразу, но оставляет структуру. Обычно быстрее, чем DELETE без условия, и часто не логируется построчно — поэтому откатить сложнее.

2. DDL — строим структуру базы

2.1 Создаём базу данных

CREATE DATABASE school_db;

Эта команда создаёт пустую базу — внутри неё мы потом сделаем таблицы.

2.2 Создаём таблицу — CREATE TABLE

Минимальный шаблон таблицы — без явного первичного ключа:

CREATE TABLE table_name (
    Attribute1 datatype,
    Attribute2 datatype,
    Attribute3 datatype
);

Пример: таблица Customer с первичным ключом:

CREATE TABLE Customer (
    CustomerID    int          NOT NULL PRIMARY KEY,
    CustomerEmail varchar(255),
    CustomerAddress varchar(255)
);

NOT NULL PRIMARY KEY означает, что CustomerID не может быть пустым и должен быть уникальным — это первичный ключ (primary key).

Пример с внешним ключом (foreign key) — таблица Item ссылается на Artist:

CREATE TABLE Item (
    ItemID   int          NOT NULL PRIMARY KEY,
    ItemDesc varchar(255),
    ItemCost real,
    ArtistID int,
    FOREIGN KEY (ArtistID) REFERENCES Artist(ArtistID)
);

Пример с составным ключом (concatenated key) — два поля вместе образуют первичный ключ:

CREATE TABLE Purchase (
    ItemID     int  NOT NULL,
    Quantity   int,
    Date       date NOT NULL,
    CustomerID int  NOT NULL,
    PRIMARY KEY (ItemID, Date, CustomerID),
    FOREIGN KEY (ItemID)     REFERENCES Item(ItemID),
    FOREIGN KEY (CustomerID) REFERENCES Customer(CustomerID)
);

2.3 Меняем таблицу — ALTER TABLE

Добавить колонку:

ALTER TABLE Customer
ADD deliveryAddress varchar(255);

Удалить колонку:

ALTER TABLE Customer
DROP COLUMN deliveryPreference;

2.4 Удаляем таблицу — DROP TABLE

DROP TABLE delivery;
⚠️ Осторожно: DROP TABLE удаляет и данные, и структуру. Восстановить можно только из резервной копии.

3. Ограничения (constraints)

Ограничения — это правила, которые СУБД сама проверяет при вставке или обновлении строки. Если правило нарушено — операция отменяется, и пользователь получает ошибку. Это важнейший инструмент целостности данных.

ОграничениеЧто гарантируетПример
PRIMARY KEY Уникально идентифицирует каждую строку. Не может быть NULL и должен быть уникальным. StudentID int PRIMARY KEY
FOREIGN KEY Поле ссылается на первичный ключ другой таблицы. Нельзя сослаться на несуществующую запись. FOREIGN KEY (ClassID) REFERENCES Class(ClassID)
NOT NULL Поле обязательно к заполнению — пусто оставить нельзя. LastName varchar(50) NOT NULL
UNIQUE Все значения в колонке должны быть разными (но допускается NULL — в отличие от PK). Email varchar(100) UNIQUE
CHECK Проверяет логическое условие. Если оно ложно — запись не сохранится. CHECK (Age >= 0 AND Age < 120)
DEFAULT Если значение не указано при вставке — подставит указанное по умолчанию. RegisteredAt date DEFAULT CURRENT_DATE

Сводная таблица школьной базы — практический пример:

CREATE TABLE Student (
    StudentID   int          NOT NULL PRIMARY KEY,
    FirstName   varchar(50)  NOT NULL,
    LastName    varchar(50)  NOT NULL,
    Email       varchar(100) UNIQUE,
    Grade       int          CHECK (Grade BETWEEN 1 AND 12),
    EnrolledAt  date         DEFAULT CURRENT_DATE,
    ClassID     int,
    FOREIGN KEY (ClassID) REFERENCES Class(ClassID)
);
Вопрос 3.1. В чём разница между PRIMARY KEY и UNIQUE? 2 балла
  • Оба гарантируют уникальность значений в колонке.
  • PRIMARY KEY + не может быть NULL + только один PRIMARY KEY на таблицу + автоматически индексируется.
  • UNIQUE + может быть NULL (одно NULL-значение допустимо в большинстве СУБД) + можно поставить на несколько колонок одновременно.
Вопрос 3.2. Зачем нужен FOREIGN KEY? Что произойдёт, если попытаться добавить запись в таблицу Order с CustomerID = 999, которого нет в таблице Customer? 3 балла
  • FOREIGN KEY обеспечивает ссылочную целостность: запись в одной таблице ссылается на существующую запись в другой.
  • Без него можно было бы записать заказ на несуществующего клиента — это бессмыслица.
  • СУБД откажет в операции и вернёт ошибку: «FOREIGN KEY constraint failed». Запись не будет вставлена.
  • Это защищает БД от «осиротевших» (orphan) записей и логических несоответствий.

4. SELECT — главный оператор

SELECT — самый важный оператор в SQL. Им мы извлекаем данные из таблиц. В IB-учебнике «запрос» (query) определяется как «инструкция на SQL для извлечения информации из БД». Запросы умеют: фильтровать (find), искать по образцу (pattern matching), сортировать данные.

4.1 Базовый синтаксис

-- Конкретные колонки
SELECT ArtistID, ArtistName FROM Artist;

-- Все колонки
SELECT * FROM Artist;
Результат — SELECT ArtistID, ArtistName FROM Artist;
ArtistIDArtistName
101Evie Ceramics
102Rojas Artwork
103Heather Needlework
104Queenies Prints
105Alejandro Jewellery
💡 Звёздочка * означает «все колонки». На практике её лучше избегать в продакшене — указывай явно нужные поля, чтобы запрос был быстрее и понятнее.

4.2 FROM — откуда брать данные

SELECT * FROM Customer;

Если данные нужны из одной таблицы — JOIN не нужен. Если из двух и более — нужен JOIN (см. раздел 9).

5. WHERE и фильтрация

5.1 WHERE — условие отбора

Аналогично оператору if в программировании — строка попадёт в результат только если условие истинно.

SELECT * FROM Item WHERE ItemCost > 300;

5.2 BETWEEN — диапазон значений

Включает оба граничных значения. Работает с числами, текстом, датами.

SELECT * FROM Item WHERE ItemCost BETWEEN 100 AND 200;
Результат — товары от 100 до 200 включительно
ItemIDItemDescItemCost

5.3 LIKE — поиск по образцу (pattern matching)

LIKE используется в WHERE для неточного совпадения. Два метасимвола:

  • _ — ровно один произвольный символ
  • % — сколько угодно символов (включая ноль)
-- Все художники, в имени которых есть слово "work"
SELECT * FROM Artist
WHERE ArtistName LIKE '%work%';
Результат — содержит «work»
ArtistIDArtistNameArtistEmail
102Rojas ArtworkRojas@mail.com
103Heather NeedleworkHNW@mail.com

5.4 IN — значение из списка

SELECT * FROM Student WHERE Grade IN (10, 11, 12);

Эквивалентно WHERE Grade = 10 OR Grade = 11 OR Grade = 12, но короче и читаемее.

5.5 Логические операторы — AND, OR, NOT

AND — все условия должны быть истинны:

SELECT itemID, ItemDesc, ItemCost
FROM Item
WHERE ItemCost > 40 AND ItemDesc LIKE '%featur%';

OR — достаточно одного условия:

SELECT * FROM Customer
WHERE CustomerCountry = 'Italy' OR CustomerCountry = 'France';

NOT — отрицание:

SELECT * FROM Customer
WHERE NOT CustomerCountry = 'Italy' AND NOT CustomerCountry = 'France';

(оставит покупателей, которые не из Италии и не из Франции.)

Вопрос 5.1. Напиши SQL-запрос: найти всех студентов 11 класса (Grade = 11) с фамилией, начинающейся на «И». 3 балла 
SELECT FirstName, LastName
FROM Student
WHERE Grade = 11
  AND LastName LIKE 'И%';

Ключевое: используем % в шаблоне, чтобы захватить любое продолжение после «И». AND объединяет оба условия.

Вопрос 5.2. Чем BETWEEN 10 AND 20 отличается от >10 AND <20? 2 балла
  • BETWEEN 10 AND 20 — включает оба значения (10 и 20).
  • >10 AND <20 — исключает оба граничных значения (только 11..19).
  • Чтобы получить тот же результат, что и BETWEEN — нужно писать >=10 AND <=20.

6. ORDER BY, DISTINCT, LIMIT, AS

6.1 ORDER BY — сортировка

По возрастанию ASC (по умолчанию) или по убыванию DESC. Можно сортировать сразу по нескольким полям.

-- От дешёвого к дорогому
SELECT * FROM Item ORDER BY ItemCost ASC;

-- Сначала по цене (от дорогого), потом по названию (по алфавиту)
SELECT * FROM Item ORDER BY ItemCost DESC, ItemDesc ASC;

6.2 DISTINCT — только уникальные значения

Используй, когда нужно убрать дубликаты. Пример: список разных товаров, которые когда-либо покупали:

SELECT DISTINCT ItemID FROM Purchase;
Результат — уникальные ItemID

6.3 LIMIT — ограничить количество строк

-- Топ-5 самых дорогих товаров
SELECT ItemDesc, ItemCost
FROM Item
ORDER BY ItemCost DESC
LIMIT 5;
💡 Заметка: LIMIT — это синтаксис SQLite, MySQL, PostgreSQL. В Microsoft SQL Server то же самое называется SELECT TOP 5 ..., в Oracle — ROWNUM <= 5. Для IB достаточно знать LIMIT.

6.4 AS — псевдонимы (алиасы)

Даём колонке или таблице временное имя — для удобства чтения результата:

SELECT FirstName AS Имя, LastName AS Фамилия
FROM Student;

-- Алиас на таблицу (для JOIN-ов)
SELECT s.FirstName, c.ClassName
FROM Student AS s
INNER JOIN Class AS c ON s.ClassID = c.ClassID;

7. Агрегатные функции

Агрегатные функции сворачивают множество строк в одно значение — это A3.3.4 на экзамене. Главное правило: их нельзя использовать в WHERE, только в SELECT и в HAVING.

ФункцияЧто считаетПример
COUNT(*)Количество строк (включая NULL-значения)Сколько всего товаров?
COUNT(col)Количество не-NULL значений в колонкеСколько товаров имеют email-адрес?
SUM(col)Сумма числовых значенийОбщая выручка
AVG(col)Среднее арифметическоеСредняя цена
MIN(col)Минимальное значениеСамый дешёвый товар
MAX(col)Максимальное значениеСамый дорогой товар

7.1 Примеры в действии — магазин Negozio

-- Сколько всего покупок?
SELECT COUNT(*) AS TotalPurchases FROM Purchase;

-- Общая выручка магазина
SELECT SUM(Quantity * ItemCost) AS Revenue
FROM Purchase
JOIN Item ON Purchase.ItemID = Item.ItemID;

-- Самый дорогой и самый дешёвый товары
SELECT MIN(ItemCost) AS Cheapest, MAX(ItemCost) AS MostExpensive
FROM Item;

-- Средний чек покупателя
SELECT AVG(ItemCost) AS AverageItemPrice FROM Item;
Результат — MIN/MAX по ItemCost

Вопрос 7.1. Напиши SQL: посчитать общее количество книг в библиотеке и среднюю цену штрафа за просрочку. 3 балла 
SELECT COUNT(*) AS TotalBooks,
       AVG(Fees)  AS AverageFine
FROM Book;

Обе функции применяются к таблице Book; AS даёт человекочитаемые названия колонок в результате.

8. GROUP BY и HAVING

8.1 GROUP BY — группируем строки

GROUP BY объединяет строки с одинаковым значением выбранного поля в одну «сводную». Применяется вместе с агрегатными функциями. 

-- Сколько покупателей в каждой стране?
SELECT COUNT(CustomerID) AS Total, CustomerCountry
FROM Customer
GROUP BY CustomerCountry;
Результат — кол-во по странам
TotalCustomerCountry
2France
2Italy
2UK

8.2 HAVING — условие для агрегата

WHERE фильтрует строки до агрегирования. HAVING фильтрует группы после агрегирования. Если в условии используется COUNT(), SUM() и др. — нужен HAVING. 

-- Страны, где у нас больше 2 покупателей
SELECT COUNT(CustomerID), CustomerCountry
FROM Customer
GROUP BY CustomerCountry
HAVING COUNT(CustomerID) > 2;
Результат — только многочисленные страны
COUNT(CustomerID)CustomerCountry
4France
3Italy
💡 Шпаргалка порядка операций в SELECT:
FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT.
Сначала откуда, потом фильтр строк, потом группировка, потом фильтр групп, потом выбор колонок, потом сортировка, потом ограничение.
Вопрос 8.1. Напиши SQL: для каждого класса в школе показать количество студентов, отсортировано по убыванию. 4 балла 
SELECT ClassID, COUNT(StudentID) AS NumStudents
FROM Student
GROUP BY ClassID
ORDER BY NumStudents DESC;

Группируем по ClassID, считаем количество студентов в каждой группе, сортируем по этому количеству в убывающем порядке.

Вопрос 8.2. Почему WHERE COUNT(*) > 5 вызовет ошибку, а HAVING COUNT(*) > 5 — нет? 3 балла
  • WHERE срабатывает до группировки — на этом этапе агрегатные функции ещё не посчитаны, поэтому СУБД не знает, что такое COUNT(*) в контексте отдельной строки.
  • HAVING срабатывает после GROUP BY — на каждую группу уже есть результат агрегата, и сравнение становится осмысленным.
  • Правило: агрегатные функции — только в HAVING и SELECT, не в WHERE.

9. JOIN — объединяем таблицы

JOIN склеивает строки из двух разных таблиц по общему полю — обычно по внешнему ключу. Это A3.3.2 на экзамене. Есть несколько типов соединений.

INNER JOIN

Только строки, где есть совпадение в обеих таблицах.

LEFT JOIN

Все строки левой таблицы + совпадения справа. Если справа нет — NULL.

RIGHT JOIN

Все строки правой таблицы + совпадения слева. Если слева нет — NULL.

FULL OUTER JOIN

Все строки из обеих таблиц. Где нет пары — NULL.

9.1 INNER JOIN — самый частый

SELECT Item.ItemID, Item.ItemDesc, Item.ItemCost,
       Artist.ArtistName, Artist.ArtistEmail
FROM Item
INNER JOIN Artist ON Item.ArtistID = Artist.ArtistID
WHERE ItemCost > 100;
Результат — товары дороже 100 + автор

9.2 LEFT JOIN — сохраняем «левых» даже без пары

Пример: показать всех студентов и их классные руководители — даже тех, кому ещё не назначили класс.

SELECT Student.FirstName, Student.LastName, Class.ClassName
FROM Student
LEFT JOIN Class ON Student.ClassID = Class.ClassID;

9.3 RIGHT JOIN — наоборот, сохраняем «правых»

-- Все классы + их студенты, даже пустые классы
SELECT Student.FirstName, Class.ClassName
FROM Student
RIGHT JOIN Class ON Student.ClassID = Class.ClassID;

9.4 FULL OUTER JOIN — всё подряд

-- Все студенты и все классы, даже не связанные
SELECT Student.FirstName, Class.ClassName
FROM Student
FULL OUTER JOIN Class ON Student.ClassID = Class.ClassID;
💡 Заметка: в SQLite (наша песочница) FULL OUTER JOIN не поддерживается напрямую — эмулируют через LEFT JOIN UNION RIGHT JOIN. В Oracle, PostgreSQL, MS SQL — работает «из коробки».

9.5 JOIN с дополнительными условиями

SELECT Purchase.ItemID, Purchase.Quantity, Purchase.Date,
       Customer.CustomerID, Customer.CustomerEmail
FROM Purchase
INNER JOIN Customer ON Purchase.CustomerID = Customer.CustomerID
WHERE Customer.CustomerCountry = 'Italy'
  AND Purchase.Date < '2025-07-01';
Вопрос 9.1. В библиотечной БД есть таблицы Borrower и Loan. Напиши запрос, который покажет имя каждого читателя и количество книг, которые он сейчас взял. 5 баллов 
SELECT Borrower.Borrower_Name,
       COUNT(Loan.Book_Number) AS BooksOnLoan
FROM Borrower
LEFT JOIN Loan ON Borrower.Borrower_ID = Loan.Borrower_ID
GROUP BY Borrower.Borrower_ID, Borrower.Borrower_Name;
  • LEFT JOIN — чтобы показать всех читателей, даже тех, у кого 0 книг.
  • GROUP BY — собираем строки по каждому читателю.
  • COUNT(Loan.Book_Number) — считает книги; для читателя без книг даст 0 (поскольку NULL не считается).
Вопрос 9.2. Чем INNER JOIN отличается от LEFT JOIN? Приведи пример, где использование INNER JOIN вместо LEFT JOIN приведёт к потере данных. 4 балла
  • INNER JOIN возвращает только строки, где совпадение есть в обеих таблицах.
  • LEFT JOIN возвращает все строки левой таблицы; если справа нет совпадения — поля справа заполняются NULL.
  • Пример: Школьная БД, таблицы Student и Mark (оценки). Запрос «показать всех студентов и их средние оценки»: 
    -- INNER JOIN потеряет студентов, у которых ещё нет оценок
SELECT Student.Name, AVG(Mark.Score)
FROM Student
INNER JOIN Mark ON Student.ID = Mark.StudentID
GROUP BY Student.ID;
    Новички без оценок исчезнут из отчёта. LEFT JOIN бы их сохранил с NULL в среднем.

10. Подзапросы (вложенный SELECT)

Подзапрос — это SELECT внутри другого SELECT (или UPDATE, DELETE). Применяют, когда нужно сначала вычислить промежуточное значение, а затем по нему отфильтровать или соединить данные.

10.1 Подзапрос в WHERE

-- Все товары, цена которых выше средней по магазину
SELECT ItemDesc, ItemCost
FROM Item
WHERE ItemCost > (SELECT AVG(ItemCost) FROM Item);

Сначала СУБД посчитает среднюю цену (одно число), потом применит его как фильтр.

10.2 Подзапрос с IN

-- Все художники, у которых хотя бы один товар был продан
SELECT ArtistName
FROM Artist
WHERE ArtistID IN (
    SELECT DISTINCT ArtistID
    FROM Item
    WHERE ItemID IN (SELECT ItemID FROM Purchase)
);

10.3 Подзапрос в FROM (производная таблица)

-- Средний чек по странам — сначала суммируем по покупателям, потом усредняем
SELECT Country, AVG(TotalSpent) AS AvgSpending
FROM (
    SELECT Customer.CustomerCountry AS Country,
           SUM(Item.ItemCost * Purchase.Quantity) AS TotalSpent
    FROM Customer
    INNER JOIN Purchase ON Customer.CustomerID = Purchase.CustomerID
    INNER JOIN Item     ON Purchase.ItemID     = Item.ItemID
    GROUP BY Customer.CustomerID
) AS PerCustomer
GROUP BY Country;
Вопрос 10.1. Напиши запрос: найти самого молодого пациента в больнице. 3 балла 
SELECT FirstName, LastName, DateOfBirth
FROM Patient
WHERE DateOfBirth = (SELECT MAX(DateOfBirth) FROM Patient);

Самый молодой — это пациент с самой поздней датой рождения (MAX по дате). Подзапрос возвращает одно значение, по которому фильтруется основная таблица.

11. INSERT, UPDATE, DELETE — изменяем данные

11.1 INSERT INTO ... VALUES — добавить запись

INSERT INTO Customer (CustomerID, CustomerEmail, CustomerAddress, deliveryAddress)
VALUES ('2', 'Christopher@mail.com', '34 Maple Crescent, Midtown',
       'Unit A Industry Road, Midtown');

11.2 INSERT INTO ... SELECT — добавить из другой таблицы

-- Скопировать всех VIP-покупателей в архивную таблицу
INSERT INTO VIPArchive (CustomerID, Email)
SELECT CustomerID, CustomerEmail
FROM Customer
WHERE TotalSpent > 10000;

11.3 UPDATE ... SET ... WHERE — изменить существующее

UPDATE Item
SET ArtistID = '101'
WHERE ItemID = '2940';

Изменили принадлежность товара 2940 другому автору.

Можно менять несколько полей сразу:

UPDATE Student
SET Grade = 11, ClassID = 22
WHERE StudentID = 105;
⚠️ Главное предупреждение: если в UPDATE забыть WHERE — все строки в таблице получат новое значение. Это самая частая ошибка с катастрофическими последствиями. 
UPDATE Item SET ItemCost = 0;   -- ОПАСНО! Все товары станут бесплатными.

11.4 DELETE FROM ... WHERE — удалить записи

DELETE FROM Item WHERE ItemID = '49393';
⚠️ Та же ловушка: DELETE FROM Item; без WHERE — удалит все строки в таблице. Сама таблица останется (это не DROP), но опустеет. Всегда сначала пиши SELECT с тем же условием, чтобы увидеть, что попадает под удаление.

Безопасная последовательность изменений

  1. Сначала SELECT с тем же WHERE — глазами проверяем, что строки правильные.
  2. Запускаем транзакцию: BEGIN TRANSACTION;
  3. Выполняем UPDATE или DELETE.
  4. Снова SELECT — убеждаемся, что изменения те, что нужно.
  5. Если всё ок — COMMIT;. Если что-то не так — ROLLBACK; (см. раздел 12).
Вопрос 11.1. В таблице Patient у пациента ID=42 сменился номер телефона на +7-916-123-4567. Напиши SQL. 2 балла 
UPDATE Patient
SET PhoneNumber = '+7-916-123-4567'
WHERE PatientID = 42;

Обязательно с WHERE по уникальному PatientID — иначе телефон сменится у всех пациентов.

Вопрос 11.2. Напиши SQL: удалить все книги, которые ни разу не выдавались (нет записей в таблице Loan). 4 балла 
DELETE FROM Book
WHERE Book_Number NOT IN (
    SELECT DISTINCT Book_Number FROM Loan
);

Подзапрос SELECT DISTINCT Book_Number FROM Loan возвращает все книги, которые когда-либо выдавались. NOT IN исключает их из удаления — остаются только «нетронутые» книги.

12. Транзакции и ACID

Это A3.3.6 на экзамене (AHL — Higher Level). Транзакция — это группа SQL-операций, которые СУБД выполняет как одно целое: либо все успешно, либо ни одной (как будто ничего не было).

12.1 BEGIN, COMMIT, ROLLBACK

BEGIN TRANSACTION;            -- начали транзакцию

UPDATE Account
SET Balance = Balance - 100
WHERE AccountID = 1;          -- сняли 100$ со счёта А

UPDATE Account
SET Balance = Balance + 100
WHERE AccountID = 2;          -- зачислили 100$ на счёт Б

COMMIT;                       -- если оба прошли — фиксируем
-- или ROLLBACK; если что-то сломалось

COMMIT «застывает» изменения навсегда. ROLLBACK восстанавливает БД к состоянию до BEGIN — отменяет всё, что произошло внутри транзакции.

⚛️
Симуляция ACID-транзакции (с инжекцией сбоя) Запусти перевод 100$ со счёта на счёт, а потом нажми «крах» — увидишь, как ROLLBACK защищает целостность.
→ Открыть ACID-симуляцию

12.2 Свойства ACID

Чтобы СУБД могла гарантировать корректность транзакций, она обеспечивает четыре свойства, аббревиатура — ACID:

A

Atomicity — Атомарность

«Всё или ничего». Если хоть одна операция в транзакции сломалась — отменяется вся транзакция. Не бывает «полупроведённой» сделки. Аналогия: перевод между счетами — нельзя списать у одного и не зачислить другому.

C

Consistency — Согласованность

БД должна перейти из одного корректного состояния в другое. Все ограничения (PRIMARY KEY, FOREIGN KEY, CHECK) после транзакции остаются выполненными. Если транзакция нарушает правило — она откатится.

I

Isolation — Изолированность

Параллельные транзакции не видят промежуточные состояния друг друга. Результат должен быть таким, как будто они шли по очереди — даже если на самом деле они шли параллельно.

D

Durability — Долговечность

После COMMIT данные сохраняются надёжно — даже если сразу после этого сервер выключат из розетки. СУБД записывает изменения в журнал на диск.

💡 Как запомнить: «Атомарно, согласованно, изолированно, долговечно». Каждое свойство решает свою проблему: A — сбои внутри транзакции; C — нарушение правил БД; I — конфликты параллелизма; D — отказы оборудования после успеха.

Классический пример — банковский перевод

Перевод 100$ со счёта Алисы на счёт Боба. Без транзакции:

  1. Снимаем 100$ у Алисы. Баланс: −100$.
  2. Сервер падает ⚡
  3. Зачисление Бобу не произошло. 100$ пропали.

С транзакцией (ACID):

  1. BEGIN TRANSACTION
  2. UPDATE Алиса −100$
  3. Сервер падает ⚡
  4. При перезапуске СУБД видит незавершённую транзакцию → автоматический ROLLBACK → баланс Алисы восстановлен.
Вопрос 12.1. Объясни, как свойство Atomicity защищает от частичного выполнения банковского перевода. 4 балла
  • Atomicity означает «всё или ничего»: транзакция выполняется как одна неделимая единица.
  • Перевод между счетами состоит из двух операций: списание и зачисление. Если выполнить только одну — деньги пропадут или появятся из ниоткуда.
  • Atomicity гарантирует, что СУБД либо применит обе операции (после COMMIT), либо отменит обе (через ROLLBACK) — например, если сервер упал после первого UPDATE.
  • В результате баланс счетов всегда консистентен: сумма денег в системе сохраняется.
Вопрос 12.2. Что делает ROLLBACK и в каких ситуациях его применяют? 3 балла
  • ROLLBACK отменяет все изменения, сделанные после последнего BEGIN TRANSACTION или COMMIT, восстанавливая БД к предыдущему стабильному состоянию.
  • Применяют когда:
    • Произошла ошибка в одной из SQL-операций (нарушение ограничения, деление на ноль и т.п.);
    • Логическая проверка показала, что результат неприемлем (например, баланс ушёл в минус);
    • Сервер аварийно перезапустился — СУБД автоматически откатит незавершённые транзакции при восстановлении.
  • Это ключевой механизм поддержания атомарности и согласованности.
Вопрос 12.3. Два кассира одновременно бронируют один и тот же последний билет на концерт. Какое свойство ACID и как это решает? 4 балла
  • Эту ситуацию решает Isolation (изолированность).
  • Без неё обе транзакции прочитали бы «свободно: 1 билет», обе списали бы его → продали два билета на одно место.
  • СУБД с изоляцией сериализует такие транзакции (например, блокировкой строки): первая транзакция увидит «1 билет», заберёт его и закоммитит; вторая увидит уже «0 билетов» и получит отказ.
  • Результат как если бы они выполнялись по очереди — несмотря на то, что физически шли параллельно.

13. Представления (VIEW) и DCL

13.1 Что такое VIEW

Представление (view) — это виртуальная таблица, основанная на сохранённом SELECT-запросе. Физически данные не дублируются; каждый раз, когда обращаемся к view, СУБД на лету выполняет лежащий внутри запрос. 

-- Создаём view: только активные товары с автором
CREATE VIEW ActiveItemsWithArtist AS
SELECT Item.ItemID, Item.ItemDesc, Item.ItemCost,
       Artist.ArtistName
FROM Item
INNER JOIN Artist ON Item.ArtistID = Artist.ArtistID
WHERE Item.Active = 1;

-- Теперь можно обращаться как к обычной таблице
SELECT * FROM ActiveItemsWithArtist WHERE ItemCost > 100;

Зачем нужны views

  • Упрощение запросов: сложный JOIN прячется за коротким именем view.
  • Безопасность: можно дать пользователю доступ только к view, а не к исходной таблице — он не увидит «лишние» колонки (например, зарплаты сотрудников).
  • Совместимость: схема меняется, а view остаётся «фасадом» с привычной структурой.

13.2 DCL — GRANT и REVOKE

GRANT выдаёт пользователю права; REVOKE их забирает.

-- Дать пользователю librarian право только читать таблицу Book
GRANT SELECT ON Book TO librarian;

-- Дать админу полные права на всю БД
GRANT ALL PRIVILEGES ON school_db.* TO admin;

-- Забрать право на удаление
REVOKE DELETE ON Book FROM librarian;
Вопрос 13.1. В чём преимущество представлений (views) с точки зрения безопасности? 3 балла
  • VIEW позволяет показать пользователю подмножество данных — только нужные колонки и строки.
  • Например, бухгалтер видит view EmployeeBasic с именами и отделами, но не видит зарплаты.
  • Сама таблица Employee при этом остаётся защищённой — на неё через GRANT доступ не выдан.
  • Это реализация принципа наименьших привилегий и сокрытия информации (information hiding).

14. Практика — пиши SQL в живой песочнице

Реально освоить SQL можно только написав много запросов руками. Открой нашу SQL-песочницу — там настоящий SQLite в браузере, с готовой школьной БД (PATIENT, DOCTOR, APPOINTMENT). Попробуй каждое из заданий ниже.

🧪
Запусти SQL-задачи прямо в браузере Любой запрос — JOIN, GROUP BY, подзапрос — пробуй и сразу видишь результат. Кнопка «reset DB» возвращает данные в исходный вид.
→ К SQL-песочнице

Задания (от простого к сложному)

  1. Покажи все колонки таблицы PATIENT.
  2. Покажи только имена и фамилии всех пациентов, отсортированных по фамилии.
  3. Найди пациентов старше 50 лет (если есть колонка возраста / даты рождения).
  4. Используя JOIN, выведи имена пациентов и фамилии их лечащих врачей.
  5. Сколько приёмов (APPOINTMENT) запланировано на каждого врача? (GROUP BY)
  6. Покажи врачей, у которых больше 3 приёмов в системе (HAVING).
  7. Используя подзапрос, найди пациента, у которого было больше всего приёмов.
  8. Создай UPDATE, который меняет специальность врача с ID=1 (не забудь WHERE!).

Что важно запомнить

  • Четыре категории SQL: DDL строит структуру, DML работает с данными, DCL управляет доступом, TCL — транзакциями.
  • Порядок предложений в SELECT: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT.
  • WHERE фильтрует строки до агрегации; HAVING — группы после агрегации. Агрегатные функции нельзя в WHERE.
  • JOIN: INNER — пересечение; LEFT/RIGHT — сохраняем «свою» сторону; FULL OUTER — всё подряд.
  • Обязательное правило безопасности: UPDATE и DELETE почти всегда требуют WHERE. Без него — катастрофа.
  • Ограничения (PRIMARY KEY, FOREIGN KEY, NOT NULL, UNIQUE, CHECK, DEFAULT) — СУБД проверяет их сама и не даст нарушить целостность данных.
  • Транзакция = группа операций как одно целое. COMMIT фиксирует, ROLLBACK откатывает.
  • ACID: Atomicity (всё или ничего), Consistency (правила не нарушаются), Isolation (без конфликтов параллелизма), Durability (после COMMIT данные надёжны).
  • Views — виртуальные таблицы, удобны для безопасности и упрощения сложных запросов.
  • На экзамене любят: «к какой категории SQL относится X?», «напиши JOIN», «объясни ACID», «зачем GROUP BY с HAVING». Всегда пиши точечно и с примером.
